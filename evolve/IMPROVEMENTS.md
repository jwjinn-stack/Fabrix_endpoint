### IMP-1 — L2→L3 드릴다운이 endpoint·namespace 차원에서 필터 없이 점프(데이터 손실)
- **Type**: ux (sev=high, effort=M)
- **Area**: `web/src/pages/Usage.tsx:182-183` (DimensionBreakdown onDrill) → `web/src/components/DimensionBreakdown.tsx:161-167`; `web/src/pages/Endpoints.tsx:325-326`; `backend/internal/server/traces.go:15`
- **Problem**: 성능 차원 분해(L2)에서 행 클릭 시 onDrill 은 `model: dim === "model" ? row.key : undefined` 로만 트레이스 필터를 시드한다. 사용자가 차원 셀렉터를 endpoint 또는 namespace 로 바꾼 뒤 튀는 행(예: 특정 엔드포인트의 TTFT 이상치)을 클릭하면, 트레이스 화면으로 이동하긴 하지만 아무 필터도 적용되지 않은 전체 목록이 뜬다. 백엔드 traces.go:15 도 `langfuse.Filters{Decision, Status, Model, App}` 만 필터하고 endpoint/namespace 트레이스 필터가 없어, 분해→증적의 핵심 워크플로(어느 그룹이 튀나 → 그 그룹의 실제 요청 보기)가 3개 차원 중 2개에서 조용히 깨진다. 행에는 여전히 '트레이스로 드릴다운' 툴팁과 clickable 커서가 붙어(DimensionBreakdown.tsx:164-166) 클릭을 유도하므로 사용자는 잘못된 결과를 정상으로 오인한다.
- **Fix**: 1순위(정공법): 백엔드 `langfuse.Filters` 에 Endpoint·Namespace 필드 추가 → traces.go 에서 `q.Get("endpoint")/q.Get("namespace")` 파싱 → Langfuse 쿼리에 반영. 프론트 `client.fetchTraces` 시그니처에 endpoint/namespace 추가. onDrill 을 차원→필터키 매핑으로 일반화(`dimFilterKey = {model:"model", endpoint:"endpoint", namespace:"namespace"}`)하여 `onNavigate("traces", { range, [dimFilterKey[dim]]: row.key })` — model 특수분기 삭제. Grafana 2026/Honeycomb BubbleUp 가 쓰는 '차원값을 필터로 보존' 패턴과 정합. 보너스: 트레이스 화면 상단에 활성 필터 칩(예: `endpoint=foo ✕`)을 보여 'L2에서 무엇을 들고 왔는지' 명시 + 한 번에 해제(가시성·되돌리기). 2순위(백엔드 지연 시 임시): 트레이스 매핑이 없는 차원에서는 행의 clickable 클래스/드릴 툴팁을 제거하고(false affordance 제거), '이 차원은 트레이스 드릴다운 미지원' 을 명시(negative affordance + 설명). 무필터 전체 목록으로의 조용한 점프는 금지.
- **Evidence**: yes (high) — 코드로 정확히 확인. traces.go:15 의 `Filters{Decision, Status, Model, App}` 에 endpoint/namespace 없음, Usage.tsx:183·Endpoints.tsx:326 의 onDrill 은 `model: dim==="model" ? row.key : undefined` 만 시드 → dim 이 endpoint/namespace 일 때 모든 필터 undefined 로 무필터 점프. ### IMP-3 — 정렬 가능한 표 헤더가 키보드로 조작 불가(WCAG 2.1.1)
- **Type**: ux (sev=medium, effort=S)
- **Area**: `web/src/components/DimensionBreakdown.tsx:147-156` (정렬 `<th>` 가 `role="button"` + onClick 만); `web/src/index.css` (`.sortable`·`.active` 규칙 없음 — grep 0건)
- **Problem**: DimensionBreakdown 표의 컬럼 헤더는 `role="button"` + onClick 으로 정렬을 토글하지만 tabIndex 가 없어 포커스를 받지 못하고 onKeyDown(Enter/Space) 핸들러도 없다 — 마우스 전용이다(WCAG 2.1.1 Keyboard 위반). index.css 에 `.sortable`/`.active` 규칙이 전혀 없어(grep 결과 0건) 정렬 가능 어포던스(커서·밑줄·정렬 화살표 색)가 약하고, 활성 정렬 컬럼 강조도 ' ↓' 글자에만 의존한다(:154). 추가로 정렬이 `cellValue(b)-cellValue(a)` 내림차순 고정(:110)이라 오름차순 토글 자체가 없다(마우스로도). 행 클릭 드릴다운(`tr onClick`, :165)도 동일하게 키보드 접근 불가.
- **Fix**: 정렬 `<th>` 의 헤더 텍스트를 실제 `<button>` 으로 감싼다(role="button"+tabIndex 보다 우선 — 네이티브 키보드/포커스/역할을 공짜로 얻음). 현재 정렬 컬럼의 `<th>` 에만 `aria-sort="ascending|descending"` 설정, 나머지는 속성 제거(none 금지). 같은 컬럼 재클릭 시 방향 토글(sortDir 상태 추가). `aria-sort` 가 macOS VoiceOver/TalkBack 에서 announce 안 되므로 폴백 2종 동반: ① 시각 화살표 ▲▼(aria-hidden) 항상 정렬 컬럼에 표시, ② `aria-live="polite"` 영역에 "정렬: {컬럼} 오름차순/내림차순" 갱신. CSS: `.sortable button{cursor:pointer}`, :hover/:focus-visible 밑줄·배경, `.active th` 에 `var(--primary)` 화살표 + `var(--primary-weak)` 컬럼 배경. ▲▼ 는 SVG 대신 문자 entity(고대비/색반전 보존). 행 드릴다운도 함께 버튼/링크화 검토.
- **Evidence**: yes (high) — DimensionBreakdown.tsx:147-153 의 정렬 `<th>` 는 role="button"+onClick 만, tabIndex·onKeyDown·aria-sort 전무. index.css grep 으로 `.sortable` 규칙 0건 확인(`.active` 는 nav/pill 등 무관 셀렉터만). 정렬 `cellValue(b)-cellValue(a)` 내림차순 고정(:110). W3C APG·Adrian Roselli·MDN 3종이 ① `<th>` 안 진짜 `<button>` 권장(네이티브 키보드 공짜), ② aria-sort 는 현재 컬럼 1개만 설정, ③ aria-sort 단독 신뢰 불가(2024-08 VoiceOver/TalkBack announce 실패) → 시각 화살표 + live region 폴백 필요 를 명시.
- **Sources**: https://www.w3.org/WAI/ARIA/apg/patterns/table/examples/sortable-table/ , https://adrianroselli.com/2021/04/sortable-table-columns.html , https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-sort
- **Deep-dive suggestion**: 없음 (출처 충분, 구현 직행)
- **Result** (2026-06-30, done): 정렬 `<th>` 를 네이티브 `<button className="th-sort">` 로 → 키보드 포커스/Enter·Space 정렬. 현재 컬럼에만 `aria-sort`(asc/desc), 방향 토글(`sortDir`), 시각 화살표 ▲▼(aria-hidden), sr-only `aria-live` 정렬 안내. index.css 에 `.th-sort` 포커스링/호버/active 추가. tsc 통과. 행 드릴다운 키보드·target-size 는 minor follow-up. spec: `specs/2026-06-30/IMP-3-sortable-header-a11y.md`.

### IMP-4 — ⓘ 정보·메트릭 의미가 hover title 툴팁에만 의존 — 키보드·터치 접근 불가(WCAG 1.4.13)
- **Type**: ux (sev=medium, effort=M)
- **Area**: `web/src/components/DimensionBreakdown.tsx:120,150,176` 및 전 페이지(grep: `title=` 34개 파일, `className="info"`); `web/src/index.css` (`.info{cursor:help}`)
- **Problem**: 메트릭 카탈로그(의미·임계치·함께 볼 메트릭)와 이상강조 사유 같은 AI grounding 의 핵심 맥락이 네이티브 `title=` 속성 툴팁으로만 제공된다(코드 전반 다수). 네이티브 title 은 키보드 포커스로 뜨지 않고, 터치 기기에서 표시되지 않으며, 등장 지연이 길고 스타일 불가다. 계층 대시보드의 가치 제안이 '메트릭을 한 출처에서 설명' 인데 그 설명이 마우스 hover 가능한 데스크톱 사용자에게만 닿는다. ⓘ 자체도 `cursor:help` span 일 뿐 button 이 아니라 포커스 불가(:120).
- **Fix**: 콘텐츠 종류로 분리해 단일 공용 시스템(기존 CSS 토큰으로 스타일)을 만든다. (1) 짧은 힌트(정렬 헤더, 간단 라벨) → hover+키보드 focus+touch 트리거 + `aria-describedby` + Esc 닫기 + hoverable·persistent(WCAG 1.4.13) + prefers-reduced-motion 존중하는 접근가능 Tooltip. (2) 풍부한 메트릭 카탈로그(의미+임계치+함께 볼 메트릭) 및 이상강조 AI-grounding 사유 → 이는 reference/essential 콘텐츠이므로 hover 툴팁 금지. TOGGLETIP/disclosure 패턴: ⓘ 를 진짜 focusable `<button>` 으로 만들어 persistent popover 토글, 또는 밀도 높은 표 셀은 기존 SlidePanel 로 라우팅. 핵심 규칙: essential/interactive 정보를 hover 뒤에 두지 말 것. 구현: 직접 만들기보다 Radix Tooltip + Radix Popover 강력 권장(headless, WAI-ARIA 1.2 정확, ~8KB each, MIT, WorkOS 인수 후 유지보수 가속). 트래픽 높은 화면(Dashboard, Traces) 먼저 점진 마이그레이션.
- **Evidence**: yes (high) — 네이티브 title 은 키보드 접근 불가·터치 미표시·SR 비일관(특히 라벨 옆 ⓘ 케이스)·지연·무스타일(Sarah Higley). WCAG 1.4.13(AA, 2018~)은 hover/focus 콘텐츠에 Dismissable/Hoverable/Persistent 를 요구하는데 native title 은 셋 다 불만족 → 진짜 AA 갭. 동일 출처가 hover 툴팁은 비필수·텍스트·비대화형만 담아야 함을 강조 → 카탈로그/AI-grounding 은 toggletip/disclosure 로. role="tooltip" 단독은 무의미, `aria-describedby` 가 load-bearing. OSS: Radix(headless+ARIA 정확, post-WorkOS 가속) 권장, Floating UI 는 positioning only, Tippy.js 는 신규 React 회피.
- **Sources**: https://sarahmhigley.com/writing/tooltips-in-wcag-21/ , https://www.w3.org/WAI/WCAG21/Understanding/content-on-hover-or-focus.html , https://www.radix-ui.com/primitives/docs/components/tooltip , https://www.pkgpulse.com/blog/floating-ui-vs-tippyjs-vs-radix-tooltip-popover-2026
- **Deep-dive suggestion**: oss-evaluate (Radix vs Floating UI vs in-house — 미니멀 스택 제약 하 비용/이점 정량화)

### IMP-5 — FABRIX MCP 서버가 백엔드에만 존재 — UI에 발견·연결·미리보기 경로가 전혀 없음
