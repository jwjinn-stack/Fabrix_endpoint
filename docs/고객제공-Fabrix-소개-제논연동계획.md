## 1. FABRIX Endpoint 개요

FABRIX Endpoint 는 **온프렘 GPU 추론 플랫폼(NVIDIA Dynamo + vLLM) 위에서 동작하는 AI 추론 거버넌스·관제 콘솔**입니다. 엔드 유저용 추론 서비스가 아니라, **운영자가 모델 서빙 현황을 관제하고, 추론 트래픽을 통제(가드레일)하며, 사용량을 부서·앱·사용자 단위로 추적하고, 증적을 불변 보존하는 관리 레이어**입니다.

핵심 설계: **모든 추론 트래픽이 Fabrix 를 단일 관문으로 통과**합니다. 따라서 가드레일 판정·사용량 귀속·증적이 누락 없이 한곳에서 일어나며, 이것이 단순 모니터링 도구와의 차별점입니다.

**5대 기능군**
| 기능군 | 한 줄 설명 |
|---|---|
| 관측(Observability) | 트래픽·추론 품질·GPU·사용량 실시간 관제 |
| 거버넌스(Governance) | PII/Jailbreak 가드레일 차단·증적·정책, WORM 불변 보존 |
| 모델 수명주기 | 레지스트리(Harbor) 임포트 → 엔드포인트 배포 → 서빙 |
| 검증(Validation) | 플레이그라운드 채팅·멀티모델 비교·LLM-as-judge 평가 |
| 운영 관리 | API 키·앱·예산, 조직·귀속, 사용자 권한(RBAC), 자격증명 |

## 2. 삼성증권 요구사항 대응 현황 (2026-06-17 미팅 기준)

> ✅ = Phase 1 구현·검증 완료 · 🟠 = 구현됐으나 보강/운영 적용 예정 · ⏳ = Phase 2

| 삼성증권 요구사항 | Fabrix 대응 기능 | 상태 |
|---|---|---|
| **서빙된 모델의 성능 모니터링** | 추론 지연 3분할(TTFT/TPOT/E2E p50·p95·p99)·생성속도(tok/s)·엔진 스케줄러(running/waiting/큐/KV캐시)·모델별 운영 메트릭(컨텍스트·GPU 요구·서빙패턴) | ✅ |
| **서비스 모니터링** | 트래픽/프록시 뷰(요청 파이프라인 분해·차단율·지연), 엔드포인트 상태(Active/Pending), 관제 대시보드 | ✅ |
| **사용자별 추론 사용량 추적** | 부서·앱·API 키·모델 4축 귀속 집계, Top5 엔드포인트/키 랭킹, 키별 토큰·요청·추정비용 | ✅ |
| (기 개발) **K8S·GPU·Pod 관제** | GPU/MIG 화면(노드 LED→GPU 테이블→드릴다운 시계열, 유휴 할당 갭 KPI), 엔드포인트(Pod) 목록·실시간 로그 | ✅ |
| **모델 추론 성능 체크 + export** | 위 성능 지표 화면 + 사용량 리포트 **CSV 내보내기**. (지표/증적 일괄 export API 보강 예정) | ✅ / 🟠 |
| **가드레일 블럭 판단 + export** | 증적 뷰(차단/통과 판정·HTTP status 403/200·지연·정책버전·사유·PII 유형), WORM 불변 보존. (증적 일괄 export 보강 예정) | ✅ / 🟠 |
| **가드레일 운영 강제(전 클라이언트)** | 현재 Fabrix 프록시 1차 강제 동작 중. 게이트웨이(Envoy ext_proc) 운영 경로 배선은 매니페스트·검증 완료, **운영 적용은 승인 대기** | 🟠 → ⏳ |
| **에이전트/챗봇 등 서비스 구성도** | 제논/삼성 제공 정보로 모니터링 항목 확장 검토 | ⏳ |
| **기본 Pod 목록 및 기능** | 엔드포인트(DynamoGraphDeployment) 목록 + 컴포넌트별 파드 로그 조회 | ✅ |

> **요지**: 삼성증권 미팅에서 나온 핵심 3개(모델 성능 모니터링·서비스 모니터링·사용자별 사용량 추적)는 **Phase 1 에서 실데이터로 구현·검증 완료**되었습니다.

## 3. 제논 환경 위에서의 연동·운영 방안

Fabrix 는 **제논 프로덕(쿠버네티스) 환경 위에 얹혀** 동작하며, 삼성생명 서비스를 위해 배포되는 모델 서빙·가드레일·관측 스택과 *읽기 위주로* 연동합니다. 제논 프로덕 자체를 대체하지 않습니다.

### 3-A. 연동 구성도 (논리)
```
[운영자 브라우저]
      │ HTTPS
      ▼
[ Fabrix Web (SPA) ] ──/api──▶ [ Fabrix BFF (Go) ]  ← 단일 추론·거버넌스 관문
                                     │
        ┌──────────┬──────────┬──────┴────┬──────────┬──────────┬──────────┐
        ▼          ▼          ▼           ▼          ▼          ▼          ▼
   메트릭         추론        가드레일      증적/사용량   불변보존    모델          오케스트레이션
 (Victoria   (Dynamo/vLLM) (Semantic    (ClickHouse) (오브젝트   레지스트리   (Kubernetes
  Metrics)    서빙 엔드포인트  Router)               스토어 WORM) (Harbor)     API)
```

### 3-B. 연동 대상(8종)과 제논 환경 매핑
> 제논 환경에 이미 존재하는 구성요소에 Fabrix 를 연결하거나, 없으면 함께 배포합니다.

| Fabrix 가 연동하는 대상 | 용도 | 제논 환경에서 확인 필요 |
|---|---|---|
| 메트릭(Prometheus 호환, 예: VictoriaMetrics) | GPU·트래픽·추론 지연 수집 | 기존 모니터링 스택 유무·쿼리 엔드포인트 |
| 추론 서빙(Dynamo/vLLM 또는 동등) | 모델 추론 호출·상태 프로브 | 서빙 플랫폼·모델 목록·OpenAI 호환 여부 |
| 가드레일 분류기(Semantic Router 등) | PII/Jailbreak 판정 | 보안 정책·분류기 사용 가능 여부 |
| 분석 DB(ClickHouse 등) | 증적·사용량 적재 | 신규 배포 가능 여부 |
| 오브젝트 스토어(MinIO/ObjectScale 등) | 증적 불변(WORM) 보존 | Object Lock 지원 스토리지 |
| 모델 레지스트리(Harbor 등) | 모델 저장·배포 원본 | 레지스트리·저장 용량 |
| 컨테이너 오케스트레이션(Kubernetes) | 엔드포인트 배포·Pod 관제 | 네임스페이스·RBAC 권한 |
| RDB(PostgreSQL 등) | 키·사용자·조직 메타 | 신규 배포 가능 여부 |

### 3-C. 운영 안전장치(거버넌스)
- **단일 관문**: 모든 추론은 가드레일 분류 후 정책에 따라 차단/통과 → 누락 없는 통제.
- **증적 불변성**: 분석 DB + 오브젝트 스토어 WORM(Object Lock, 기본 보존 365일) 이중 보존 — 변조 불가(규제/감사 대응).
- **비식별화**: 증적의 사용자 식별자는 솔트 해시, 프롬프트 원문/PII 미저장(마스킹 샘플만).
- **운영 리소스 보호**: 엔드포인트 변경/삭제는 Fabrix 가 생성한 리소스로 한정, 운영 네임스페이스는 보호.
- **API 키**: 원문 미저장(해시+prefix), 발급 시 1회 노출. 분당 요청·일 토큰 예산 하드캡(초과 시 자동 차단).

## 4. 개발 로드맵 (Phase 1 / Phase 2)

> 임재황 책임 정리 기준: **Phase 1 ≈ 3개월 + Phase 2 ≈ 3개월 이내**, 고객 요청에 따라 기간 조율.

### Phase 1 — 기반 관제·거버넌스·모델 수명주기 (구현·검증 완료)
- 관제 대시보드(트래픽·품질·GPU·가드레일 요약), GPU/MIG 관제(드릴다운·유휴 갭)
- 가드레일(PII/Jailbreak/Secrets) 차단·증적·정책 카탈로그·WORM 불변 보존
- 모델 레지스트리(Harbor) 임포트(HF/NGC/업로드) → 엔드포인트 배포 위저드 → 서빙
- 플레이그라운드(채팅·멀티모델 비교·코드 보기), LLM-as-judge 평가
- 사용량·귀속(부서/앱/키/모델, CSV export), API 키·예산(쿼터·하드캡), RBAC/사용자
- **추론 성능 모니터링**(TTFT/TPOT/E2E·tok/s·스케줄러), **Top5 랭킹**, **모델 운영 메트릭**, **실시간 Pod 로그**

### Phase 2 — 운영 강제·심화 관측·자동화 (계획)
- 가드레일 **게이트웨이 운영 강제**(Envoy ext_proc — 모든 클라이언트 강제)
- **개별 요청 분산 트레이스**(gateway→router→engine 스팬 트리, OTLP 수집)
- **MIG 슬라이스 단위 관제**(MIG 파티션 활성화 시), **scale-to-zero·SLO Planner**
- 평가 **데이터셋·회귀 배치**, 응답별 TTFT(스트리밍), 예산/경고 임계 **영속화**
- **외부 사내 디렉터리 연동**(세션ID→직원 매핑으로 사용자별 귀속 정밀화)
- 증적·성능 지표 **일괄 export API**(삼성 요구 대응 보강), 에이전트/챗봇 서비스 모니터링 확장
- 인클러스터 정식 배포 활성화(이미지 레지스트리 인증·매니페스트 적용)

## 5. 제논 측에 요청드리는 정보 (연동 설계 입력)

> 세부 연동 방안 확정을 위해 아래 정보를 요청드립니다(임재황 책임 정리 기준).

1. **환경 정보** — 마스터 노드 및 GPU/CPU 노드 구성 현황과 사양
2. **서비스 정보** — 제논 프로덕 자체를 제외하고, 삼성생명 서비스를 위해 배포되는 서비스(모델 서빙/가드레일 구성, 엔드 유저용 서비스 등)
3. **모델 서빙 현황** — 개별 모델 스펙 및 서빙 플랫폼
4. **모델 추론 성능 체크** — 성능 측정 기능 보유 여부 및 데이터 export 가능 여부
5. **가드레일** — 블럭 판단 여부 확인 가능성 및 판정 데이터 export 가능 여부
6. **서비스 구성도** — 에이전트/챗봇 등 서비스 구성(모니터링 항목 추가 검토용)
7. **Pod 목록** — 기본 Pod 목록 및 각 기능

> ※ 라이선스 정책은 영업 검토 후 별도 회신 예정입니다.

## 6. 검증 현황 및 알려진 제약 (사실 기재)

**검증 완료(Phase 1)**: 실추론(가드레일 통과·차단)·증적 적재·부서/앱/키 귀속·키 예산 하드캡(429)·모델 임포트(HF→레지스트리)·엔드포인트 배포(레지스트리 참조→서빙)·자격증명 저장·GPU/추론 성능 지표 표시.

**알려진 제약**
- **대용량 모델 임포트 보류**: 모델 레지스트리의 블롭이 오브젝트 스토어에 저장되는 구조에서, 특정 스토리지 노드의 디스크 여유 부족으로 대용량 모델(예 ~59GB) 임포트가 보류 중입니다. 소형 모델은 정상. 스토리지 용량 확보 후 진행하며, 코드/기능 자체는 정상입니다(상세 [인프라 §10](인프라-통신-아키텍처.md)).
- **조직·귀속 화면 점검 중**: 런타임 사용자 귀속을 앱-부서 기준으로 일원화하는 보정 작업 진행 중.
- **가드레일 게이트웨이 운영 강제**: 매니페스트·검증 완료, 운영 경로 적용은 환경 승인 대기(현재 프록시 1차 강제 동작 중).
