# 쿠버 배포 설정 레퍼런스 (정방향)

> **"이 화면([연동 상태](../../web/src/pages/Diagnostics.tsx))을 초록으로 만들려면 매니페스트에 무엇을 어떻게 넣어야 하나."**
> 같은 바이너리를 env 만으로 두 제품처럼 배포한다 → **observe**(읽기 전용 관제, 예: 삼성증권) / **manage**(엔드포인트 생성·삭제까지 풀버전).
> 문제 추적(역방향)은 [통신디버깅-런북.md](./통신디버깅-런북.md), 설계 추적은 [연동상태-통신디버깅-개선-추적표.md](./연동상태-통신디버깅-개선-추적표.md).

코드 출처: 프로파일 해석 [capability.go](../../backend/internal/capability/capability.go) · env 로드 [config.go](../../backend/internal/config/config.go) · 프로브 [diagnostics.go](../../backend/internal/server/diagnostics.go).

## 1. 두 프로파일 = 한 바이너리

| | **observe** (읽기 전용) | **manage** (풀버전) |
|---|---|---|
| `FABRIX_PROFILE` | `observe` | `manage` (기본) |
| 켜지는 기능 | dashboard · traces · guard · models (전부 read) | 전부 (생성·변경·삭제 포함) |
| mutating 라우트 | **등록 안 됨**(404 — 공격 표면 제거) | 등록됨 |
| 미세조정 | `FABRIX_FEATURES="+endpoints,+keys"` 로 읽기 화면 추가 | 보통 불필요 |

> observe 는 mutating cap 이 0이라 `Readonly()=true`. 라우트 자체가 미등록이라 "권한 없음"이 아니라 **존재하지 않음**으로 막는다.

## 2. 프로파일별 필요 의존성 매트릭스

각 의존성이 어떤 기능(capability)을 받치는지 → 그 기능이 프로파일에 켜져 있으면 그 의존성이 필요하다.

| 의존성 | env | 받치는 기능 | **observe** | **manage** | 미설정 시 폴백 |
|---|---|---|---|---|---|
| 메트릭 (VictoriaMetrics) | `FABRIX_VMSELECT_URL` (+`FABRIX_DATA_SOURCE=live`) | dashboard | 필요¹ | 필요¹ | mock 합성 메트릭 |
| 추론 업스트림 (Dynamo) | `FABRIX_GEMMA_UPSTREAM` | models·playground | 필요(모델 상태) | 필요 | 모델 unreachable 표시 |
| 가드레일 (Semantic Router) | `FABRIX_SR_URL` | guard·guard.write | 선택² | 필요 | 모든 요청 통과(차단 없음) |
| 증적 (ClickHouse) | `FABRIX_CLICKHOUSE_URL` | guard | 필요 | 필요 | 증적 비적재(판정은 동작) |
| 사용량 롤업 (ClickHouse) | `FABRIX_CLICKHOUSE_URL` | dashboard | 필요 | 필요 | 롤업/추세 비적재 |
| 트레이스 (Langfuse) | `FABRIX_LANGFUSE_HOST`·`_PUBLIC_KEY`·`_SECRET_KEY` | traces | 필요³ | 필요³ | synthetic 트레이스(화면 안 빔) |
| 키 스토어 (PostgreSQL) | `FABRIX_DATABASE_URL` | keys·users | **불요**⁴ | 필요 | 키·앱·사용자 기능 비활성 |
| 모델 레지스트리 (Harbor) | `FABRIX_HARBOR_URL` | models·models.write | 필요(읽기) | 필요 | 모델 목록/임포트 비활성 |
| 오케스트레이션 (K8s) | in-cluster RBAC / `FABRIX_KUBECTL`·`FABRIX_ENDPOINTS_NS` | endpoints | **불요**⁴ | 필요 | 엔드포인트 조회/배포 비활성 |
| WORM 보존 (MinIO) | `FABRIX_WORM_URL`·`_BUCKET`·`_RETAIN_DAYS` | guard | 선택 | 선택 | ClickHouse 증적만(불변 보존 없음) |

¹ `FABRIX_DATA_SOURCE=mock`(기본)이면 vmselect 없이 합성 메트릭으로 동작. 실연동은 `live` + URL.
² observe 의 guard 는 **증적 조회(ClickHouse)** 라 SR 연결이 필수는 아님. 정책 enforcement/분류 테스트(guard.write)는 manage 전용.
³ Langfuse 미설정이어도 synthetic 폴백으로 화면은 동작 → 실 트레이스가 필요할 때만 구성.
⁴ observe 는 keys/users/endpoints 가 꺼져 있어 **아예 필요 없음**. `FABRIX_FEATURES`로 읽기만 켜면 그때 필요.

## 3. env 전체 레퍼런스 (어디에 넣나)

**자격증명이 섞인 URL 은 Secret, 나머지는 ConfigMap.** (creds 는 화면에서 항상 마스킹되지만 매니페스트에선 분리해야 한다.)

### ConfigMap (비밀 아님)
| env | 값 예시(인클러스터 FQDN) | 비고 |
|---|---|---|
| `FABRIX_PROFILE` | `observe` / `manage` | 프로파일 |
| `FABRIX_FEATURES` | `+endpoints,+keys` | observe 미세조정(선택) |
| `FABRIX_API_ADDR` | `:8080` | 리슨 주소 |
| `FABRIX_ALLOWED_ORIGINS` | `https://fabrix.samsung.example` | CORS |
| `FABRIX_DATA_SOURCE` | `live` / `mock` | 메트릭 소스 |
| `FABRIX_VMSELECT_URL` | `http://vmselect-vm.observability:8481/select/0/prometheus` | |
| `FABRIX_GEMMA_UPSTREAM` | `http://gemma4-31b-vllm-agg-frontend-nodeport.dynamo-inference:8000` | |
| `FABRIX_SR_URL` | `http://semantic-router.vllm-semantic-router-system:8080` | manage |
| `FABRIX_LANGFUSE_HOST` | `http://langfuse-web.langfuse.svc.cluster.local:3000` | |
| `FABRIX_KUBECTL` | (인클러스터는 비움 → in-cluster API) | manage |
| `FABRIX_ENDPOINTS_NS` | `dynamo-inference` | manage |
| `FABRIX_WORM_BUCKET` | `fabrix-worm` | 선택 |
| `FABRIX_WORM_RETAIN_DAYS` | `365` | 선택 |
| `FABRIX_POLICY_VERSION` | `v1` | 증적 정책 버전 |

### Secret (자격증명 포함)
| env | 값 형태 | 비고 |
|---|---|---|
| `FABRIX_CLICKHOUSE_URL` | `http://fabrix:<pw>@clickhouse.fabrix-endpoint:8123` | 증적·사용량 |
| `FABRIX_DATABASE_URL` | `postgres://fabrix:<pw>@fabrix-pg-rw.fabrix-endpoint:5432/fabrix` | manage |
| `FABRIX_HARBOR_URL` | `https://admin:<pw>@harbor-core.harbor-system.svc.cluster.local` | |
| `FABRIX_WORM_URL` | `https://<key>:<secret>@minio.fabrix-endpoint:9000` | 선택 |
| `FABRIX_LANGFUSE_PUBLIC_KEY` | `pk-lf-...` | |
| `FABRIX_LANGFUSE_SECRET_KEY` | `sk-lf-...` | |
| `FABRIX_AUDIT_SALT` | 랜덤 문자열 | user_ref 비식별 해시 솔트 |

> **DNS 팁**: 다른 네임스페이스의 서비스는 단일 라벨(`clickhouse`)이 아니라 FQDN(`clickhouse.fabrix-endpoint.svc.cluster.local`)로 적으면 search domain 의존이 사라져 `dns_fail` 을 예방한다. 화면 "네트워크·설정 점검"의 해석 테이블이 이걸 즉시 검증해준다.

## 4. observe 매니페스트 스켈레톤 (읽기 전용)

> 최소 의존성: 메트릭·증적·트레이스·모델 읽기. **PostgreSQL·K8s RBAC·WORM 불요.**

```yaml
apiVersion: v1
kind: ConfigMap
metadata: { name: fabrix-config, namespace: fabrix-endpoint }
data:
  FABRIX_PROFILE: "observe"
  FABRIX_DATA_SOURCE: "live"
  FABRIX_ALLOWED_ORIGINS: "https://fabrix.samsung.example"
  FABRIX_VMSELECT_URL: "http://vmselect-vm.observability:8481/select/0/prometheus"
  FABRIX_GEMMA_UPSTREAM: "http://gemma4-31b-vllm-agg-frontend-nodeport.dynamo-inference:8000"
  FABRIX_LANGFUSE_HOST: "http://langfuse-web.langfuse.svc.cluster.local:3000"
---
apiVersion: v1
kind: Secret
metadata: { name: fabrix-secrets, namespace: fabrix-endpoint }
stringData:
  FABRIX_CLICKHOUSE_URL: "http://fabrix:CHANGEME@clickhouse.fabrix-endpoint:8123"
  FABRIX_HARBOR_URL: "https://admin:CHANGEME@harbor-core.harbor-system.svc.cluster.local"
  FABRIX_LANGFUSE_PUBLIC_KEY: "pk-lf-CHANGEME"
  FABRIX_LANGFUSE_SECRET_KEY: "sk-lf-CHANGEME"
  FABRIX_AUDIT_SALT: "CHANGEME-random"
---
apiVersion: apps/v1
kind: Deployment
metadata: { name: fabrix-endpoint, namespace: fabrix-endpoint }
spec:
  replicas: 2
  selector: { matchLabels: { app: fabrix-endpoint } }
  template:
    metadata: { labels: { app: fabrix-endpoint } }
    spec:
      # observe 는 K8s API 조작이 없어 default SA 로 충분
      containers:
        - name: api
          image: harbor-core/fabrix/endpoint:latest
          ports: [{ containerPort: 8080 }]
          envFrom:
            - configMapRef: { name: fabrix-config }
            - secretRef: { name: fabrix-secrets }
          readinessProbe:
            httpGet: { path: /api/v1/healthz, port: 8080 }
          livenessProbe:
            httpGet: { path: /api/v1/healthz, port: 8080 }
---
# egress 화이트리스트(default-deny 환경). DNS + 읽는 의존성 포트만 연다.
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: fabrix-egress, namespace: fabrix-endpoint }
spec:
  podSelector: { matchLabels: { app: fabrix-endpoint } }
  policyTypes: [Egress]
  egress:
    - to: [{ namespaceSelector: { matchLabels: { kubernetes.io/metadata.name: kube-system } } }]
      ports: [{ protocol: UDP, port: 53 }, { protocol: TCP, port: 53 }]   # CoreDNS
    - {} # 단순화: 전체 egress. 보안 강화 시 아래로 교체
    # - to: [...vmselect 8481, clickhouse 8123, langfuse 3000, dynamo 8000, harbor 443...]
```

## 5. manage 매니페스트 추가분 (풀버전)

observe 위에 **PostgreSQL·SR·K8s RBAC·(선택)WORM** 을 더한다.

```yaml
# ── ConfigMap 추가 ──
  FABRIX_PROFILE: "manage"
  FABRIX_SR_URL: "http://semantic-router.vllm-semantic-router-system:8080"
  FABRIX_ENDPOINTS_NS: "dynamo-inference"
  FABRIX_WORM_BUCKET: "fabrix-worm"        # 선택
  FABRIX_WORM_RETAIN_DAYS: "365"           # 선택
  # FABRIX_KUBECTL 은 비움 → in-cluster API 사용
---
# ── Secret 추가 ──
  FABRIX_DATABASE_URL: "postgres://fabrix:CHANGEME@fabrix-pg-rw.fabrix-endpoint:5432/fabrix"
  FABRIX_WORM_URL: "https://KEY:SECRET@minio.fabrix-endpoint:9000"   # 선택
---
# ── 엔드포인트 오케스트레이션용 ServiceAccount + RBAC ──
apiVersion: v1
kind: ServiceAccount
metadata: { name: fabrix-endpoint, namespace: fabrix-endpoint }
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role          # 엔드포인트(DynamoGraphDeployment CR)를 다루는 네임스페이스
metadata: { name: fabrix-orchestrator, namespace: dynamo-inference }
rules:
  - apiGroups: ["nvidia.com"]              # DynamoGraphDeployment CRD 그룹(환경에 맞게)
    resources: ["dynamographdeployments"]
    verbs: ["get","list","watch","create","delete"]
  - apiGroups: [""]
    resources: ["pods","pods/log"]
    verbs: ["get","list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: { name: fabrix-orchestrator, namespace: dynamo-inference }
subjects:
  - kind: ServiceAccount
    name: fabrix-endpoint
    namespace: fabrix-endpoint
roleRef: { kind: Role, name: fabrix-orchestrator, apiGroup: rbac.authorization.k8s.io }
```

Deployment 의 `spec.template.spec` 에 `serviceAccountName: fabrix-endpoint` 를 추가하고,
NetworkPolicy egress 에 PostgreSQL(5432)·SR(8080)·K8s API(443)·MinIO(9000) 포트를 더한다.

> **K8s 프로브 실패(`auth_fail`) 시**: `kubectl auth can-i --as=system:serviceaccount:fabrix-endpoint:fabrix-endpoint get dynamographdeployments -n dynamo-inference` 로 RBAC 확인.

## 6. 검증 루프 (배포 → 화면 → 고침)

```bash
# 1) 배포
kubectl apply -f fabrix-<profile>.yaml

# 2) 화면(또는 API)으로 진단 — 이 Pod 안에서 실제 통신
kubectl -n fabrix-endpoint exec deploy/fabrix-endpoint -- \
  wget -qO- 'localhost:8080/api/v1/diagnostics?verbose=1' | jq '.summary, .network.proxy_warnings, (.checks[]|select(.configured and (.reachable|not))|{name,fail_kind,error})'

# 3) 빨강이면 fail_kind 로 고칠 곳 판단 → 런북 §1 표 → 매니페스트 수정
# 4) 반영
kubectl -n fabrix-endpoint rollout restart deploy/fabrix-endpoint
# 5) 화면 "재검사" → degraded 0 될 때까지 반복
```

**판정 기준**: `summary.degraded == 0` (구성된 의존성 전부 도달). 미구성(폴백)은 의도된 것이면 정상.
