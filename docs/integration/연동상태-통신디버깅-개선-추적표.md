# 연동 상태 → 통신 디버깅 콘솔 전환 — 구현 추적표

> 현재 `연동 상태`(`/diagnostics`) 화면은 의존성별 `초록/빨강 + latency_ms` 1비트만 보여준다.
> 이 문서는 그것을 **"파드에서 외부로 나가는 통신·네트워크·설정이 실제로 어떻게 동작/실패하는지 단계별로 들여다보는 디버깅 콘솔"** 로 끌어올리기 위한 작업 추적표다.
> 실사례(Go `httptrace`, Spring Actuator `details`, ASP.NET HealthChecks UI, Datadog flare, K8s 네트워크 진단)를 우리 코드에 매핑한다.

- 대상 화면: [web/src/pages/Diagnostics.tsx](../../web/src/pages/Diagnostics.tsx)
- 대상 API: `GET /api/v1/diagnostics` → [backend/internal/server/diagnostics.go](../../backend/internal/server/diagnostics.go)
- 핵심 모델: [backend/internal/diag/diag.go](../../backend/internal/diag/diag.go)
- 상태 범례: ⬜ 미착수 · 🟦 진행중 · ✅ 완료 · ⏸ 보류

## 0. 한눈에 보기 — 4개 진단 레이어

| 레이어 | 답하려는 질문 | 현재 | 목표 |
|---|---|---|---|
| **L1 도달성** | 연결됐나? | ✅ `reachable` / `latency_ms` | 유지 |
| **L2 단계 타이밍** | 어디서 느리고 어디서 끊겼나? (DNS·TCP·TLS·서버) | ❌ 단일 latency | `Timing{}` + 단계 바 |
| **L3 통신 증거** | 무엇을 보냈고 무엇이 왔나? (status·헤더·본문·TLS 인증서) | ❌ | `ProbeTrace{}` (verbose 모드, 마스킹) |
| **L4 설정/네트워크** | 설정값·DNS·프록시·인클러스터가 올바른가? | ⚠️ endpoint만 | `Network{}` + `Config{}` 섹션 |

핵심 인사이트: 현재 모든 HTTP 프로브가 `http.NewRequestWithContext(ctx, …)`로 `ctx`를 그대로 넘긴다 →
`diag.Run`에서 `ctx`에 `httptrace.ClientTrace`를 주입하면 **프로브 코드 변경 없이** L2 타이밍이 자동 수집된다.

## 2. 목표 데이터 모델 (`diag.go`)

```go
// Status 확장 (기존 필드 유지 + 아래 추가)
type Status struct {
    // ...기존: Name, Title, Category, Endpoint, Configured, Reachable, LatencyMs, Error,
    //          Optional, RequiredBy, FallbackNote ...

    FailKind string         `json:"fail_kind,omitempty"` // dns_fail|conn_refused|tls_fail|auth_fail|timeout|bad_status|ok
    Timing   *Timing        `json:"timing,omitempty"`    // L2 단계 타이밍
    Details  map[string]any `json:"details,omitempty"`   // L3 증거 데이터(Actuator details 패턴)
    Probe    *ProbeTrace    `json:"probe,omitempty"`     // L3 요청/응답 캡처(verbose 모드만)
    History  []Sample       `json:"history,omitempty"`   // L4 최근 N회 추세(sparkline)
}

type Timing struct { // ms 단위, 미측정 단계는 0
    DNSMs     int64 `json:"dns_ms"`      // 이름 해석
    ConnectMs int64 `json:"connect_ms"`  // TCP 3-way
    TLSMs     int64 `json:"tls_ms"`      // TLS 핸드셰이크
    TTFBMs    int64 `json:"ttfb_ms"`     // 첫 응답 바이트까지(서버 처리 포함)
    ServerMs  int64 `json:"server_ms"`   // TTFB - (DNS+Connect+TLS) = 순수 서버 처리 추정
    TotalMs   int64 `json:"total_ms"`
    Reused    bool  `json:"reused"`      // keep-alive 커넥션 재사용 여부(GotConnInfo.Reused)
}

type ProbeTrace struct {
    Method      string            `json:"method"`
    URL         string            `json:"url"`           // Redact 적용
    ReqHeaders  map[string]string `json:"req_headers"`   // Authorization → "Bearer ***"
    StatusCode  int               `json:"status_code"`
    HTTPVersion string            `json:"http_version"`
    RespSnippet string            `json:"resp_snippet"`  // 앞 512B, 스크럽
    TLS         *TLSInfo          `json:"tls,omitempty"`
}

type TLSInfo struct {
    Version   string `json:"version"`     // "TLS 1.3"
    Cipher    string `json:"cipher"`
    Subject   string `json:"subject"`     // 인증서 CN/SAN
    Issuer    string `json:"issuer"`
    NotAfter  string `json:"not_after"`   // 만료일 — 사내 인증서 만료 디버깅
    DaysLeft  int    `json:"days_left"`
}

type Sample struct {
    At        string `json:"at"`
    Reachable bool   `json:"reachable"`
    LatencyMs int64  `json:"latency_ms"`
}

// Network 는 파드 레벨 네트워크/설정 점검(의존성 무관, 1회).
type Network struct {
    InCluster      bool              `json:"in_cluster"`      // serviceaccount 토큰 존재
    KubeDNS        string            `json:"kube_dns"`        // /etc/resolv.conf nameserver
    SearchDomains  []string          `json:"search_domains"`  // resolv.conf search
    HTTPProxy      string            `json:"http_proxy"`      // env (값 마스킹)
    NoProxy        string            `json:"no_proxy"`
    ProxyWarnings  []string          `json:"proxy_warnings"`  // 업스트림 호스트가 NO_PROXY에 없음 등
    DNS            []DNSCheck        `json:"dns"`             // 의존성 호스트별 이름 해석 결과
}

type DNSCheck struct {
    Host      string   `json:"host"`
    Resolved  []string `json:"resolved"`  // IP 목록
    LatencyMs int64    `json:"latency_ms"`
    Error     string   `json:"error,omitempty"`
}

// Report 에 Network 추가
type Report struct {
    GeneratedAt string  `json:"generated_at"`
    Profile     string  `json:"profile"`
    Summary     Summary `json:"summary"`
    Network     *Network `json:"network,omitempty"`
    Checks      []Status `json:"checks"`
}
```

## 3. 핵심 기법별 구현 메모

### B1. httptrace 단계 타이밍 (프로브 무수정)
`diag.Run`의 고루틴에서 probe 호출 전, ctx에 trace를 심는다:

```go
import "net/http/httptrace"

go func(i int, probe func(context.Context) error, st Status) {
    defer wg.Done()
    var t Timing
    var dnsStart, connStart, tlsStart, start time.Time
    start = time.Now()
    trace := &httptrace.ClientTrace{
        DNSStart:  func(httptrace.DNSStartInfo) { dnsStart = time.Now() },
        DNSDone:   func(httptrace.DNSDoneInfo)  { t.DNSMs = ms(dnsStart) },
        ConnectStart: func(_, _ string) { connStart = time.Now() },
        ConnectDone:  func(_, _ string, _ error) { t.ConnectMs = ms(connStart) },
        TLSHandshakeStart: func() { tlsStart = time.Now() },
        TLSHandshakeDone:  func(cs tls.ConnectionState, _ error) { t.TLSMs = ms(tlsStart); /* TLSInfo 추출 */ },
        GotConn:   func(g httptrace.GotConnInfo) { t.Reused = g.Reused },
        GotFirstResponseByte: func() { t.TTFBMs = time.Since(start).Milliseconds() },
    }
    pctx := httptrace.WithClientTrace(ctx, trace)
    err := probe(pctx)              // ← 기존 프로브가 pctx를 그대로 NewRequestWithContext에 전달
    t.TotalMs = time.Since(start).Milliseconds()
    t.ServerMs = max(0, t.TTFBMs-(t.DNSMs+t.ConnectMs+t.TLSMs))
    st.Timing = &t
    st.FailKind = classify(err, &t)  // B2
    // ...기존 LatencyMs/Error/Reachable 세팅...
}(...)
```

> 주의: pgx(`postgresql`)·kubectl(`kubernetes`) 프로브는 httptrace 비대상 → `Timing`은 HTTP 프로브에만 채워지고 나머지는 `nil`. MinIO(`worm`)는 minio-go가 내부적으로 `http.Client`라 ctx 전파 시 부분 수집 가능(검증 필요).

### B2. 실패 원인 분류 (`classify`)
에러 문자열 + 도달 단계로 사용자 행동이 다른 6종 분기:

| FailKind | 판정 단서 | 사용자 행동 | 배지 색 |
|---|---|---|---|
| `dns_fail` | `no such host`, DNSDone 미도달 | 서비스명 오타 / CoreDNS / search domain | red |
| `conn_refused` | `connection refused`, ConnectDone 실패 | 포트·NetworkPolicy egress·Pod 미기동 | red |
| `tls_fail` | `tls:`, `x509`, cert 만료 | 인증서·CA·SNI·프록시 MITM | red |
| `auth_fail` | status 401/403 | 키·시크릿·RBAC | amber |
| `timeout` | `context deadline`, `i/o timeout` | 방화벽 drop·과부하·잘못된 IP | red |
| `bad_status` | status 5xx (도달O) | 업스트림 자체 장애 | amber |
| `ok` | err==nil | — | green |

### B3. Details 페이로드 (Actuator 패턴)
프로브 시그니처를 `func(ctx) (map[string]any, error)`로 진화시키거나, 클라이언트에 `ProbeDetails(ctx)` 추가.
- Harbor: `{harbor_version, projects_visible}` (이미 [Status()](../../backend/internal/harbor/harbor.go#L144)가 비슷한 데이터 생산 → 재사용)
- ClickHouse: `{server_version, query:"SELECT 1", rows:1}`
- Langfuse: `{api_version, traces_returned:1}`
- K8s: `{healthz:"ok", context:"in-cluster", ns}`

### B4. 증거 캡처 + 마스킹 (Datadog flare 패턴)
- verbose 모드(`?verbose=1` 또는 카드별 버튼)에서만 `ProbeTrace` 채움 — 상시 캡처 금지(부하·보안).
- [Redact()](../../backend/internal/diag/diag.go#L114) 원칙 확장: `Authorization`/`X-*-Key`/`Cookie` → `***`, 본문은 앞 512B만 + 토큰 패턴 스크럽.

### B5. 파드 네트워크/설정 점검 (`Network{}`)
- **DNS**: 각 의존성 호스트에 `net.Resolver.LookupHost(ctx, host)` → 해석 IP·지연. `no such host`면 L4에서 미리 잡음(프로브 던지기 전 90% 원인).
- **resolv.conf**: `/etc/resolv.conf` 파싱 → nameserver(kube-dns 10.96.0.10 등)·search domain.
- **in-cluster**: `/var/run/secrets/kubernetes.io/serviceaccount/token` 존재 + `KUBERNETES_SERVICE_HOST`.
- **proxy**: `HTTP(S)_PROXY`/`NO_PROXY` env 읽고, 인클러스터 업스트림 호스트가 `NO_PROXY`에 없으면 경고(프록시 경유로 사내 svc 못 감 — 흔한 함정).
- **설정 해석(Config)**: 각 `FABRIX_*` env → 파싱된 `{scheme, host, port}`(마스킹) 표시. 스킴 누락·포트 오타를 노출.

### B6. 경량 history (HealthChecks UI 패턴)
- `Server`에 `map[string]*ring(50)` 보관(외부 저장소 불필요). `diag.Run` 후 각 결과를 push, 응답에 `History` 동봉.
- 프론트는 latency sparkline + "마지막 정상/실패 시각".

## 4. 구현 추적표 (메인)

> **구현 완료 (2026-06-28).** 아래 표의 상태를 갱신. 두 가지 설계 변경:
> - **B0 폴딩**: 별도 `httpx/probe.go` 없이 `diag.Run`이 ctx 에 httptrace 를 주입하는 방식으로 구현 → 프로브 클라이언트 무수정으로 Timing/TLS 수집(더 단순·안전).
> - **B4 축소**: 요청/응답 **본문/헤더 캡처는 보류**. 대신 httptrace 로 확실히 얻는 증거(연결 IP·단계 타이밍·TLS 인증서·연결 재사용)와 에러 파싱 status 를 항상 노출. 본문 캡처는 클라이언트별 transport 래핑이 필요해 향후 과제로 분리(아래 "남은 과제").
