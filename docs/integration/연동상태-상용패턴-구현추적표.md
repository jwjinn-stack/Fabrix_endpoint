# 연동 상태 → 상용 패턴 차용 구현 추적표

> 상용 레퍼런스(AWS Reachability Analyzer · Kiali · ArgoCD · Komodor · Stripe · Airbyte · Spring Boot Admin)의 패턴을
> **우리 실제 코드와 대조**해, 이미 데이터가 있어 바로 되는 것부터 능동 제어까지 **실현 가능한 순서**로 정리한다.
> 원칙: ① 앱 단독으로 가능할 것(CNI/클라우드 API 연동 불요) ② 기존 필드 재사용 우선 ③ 못/안 할 것은 §4에 정직하게.

대상: [Diagnostics.tsx](../../web/src/pages/Diagnostics.tsx) · [diag.go](../../backend/internal/diag/diag.go) · [network.go](../../backend/internal/diag/network.go) · [diagnostics.go](../../backend/internal/server/diagnostics.go)

## 3. 권장 순서 (마일스톤)

각 마일스톤은 독립 출시 가능. 위로 갈수록 ROI 대비 비용 낮음.

| M | 범위 | 포함 | 결과 |
|---|---|---|---|
| **M1 — 화면 표현(백엔드 0)** | 이미 있는 데이터를 더 잘 보이게 | **C1·C2·C6·C7** | 경로 스테퍼·영향범위·자동폴링·이력 — 백엔드 변경 없이 디버깅 체감 ↑ |
| **M2 — 설정 검증** | 배포 전 오류를 프로브 전에 잡기 | **C3·C4** | env 린트 + "이 프로파일에서 뭐가 빠졌나"(ArgoCD식) |
| **M3 — 능동 제어** | manage 에서 직접 테스트 | **C5** | dep별 "지금 쏴보기"(Stripe식), write cap 게이팅 |
| **M4 — 심화** | 추세·변경·드릴다운 | **C8·C9** | 설정 변경 감지 + 컴포넌트 details |

**시작점: M1 (특히 C1·C2)** — 코드 변경이 [Diagnostics.tsx](../../web/src/pages/Diagnostics.tsx) 안에서 끝나고(백엔드 0), 디버깅 체감이 가장 큼. 그다음 M2 의 C3(설정 린트)가 "배포 시 미리 잡기"로 직결.

## 4. 범위 밖 (비권장 — 정직하게)

실현 가능성을 지키려면 **안 하는 것**도 명확히.

| 안 하는 것 | 이유 | 대체 |
|---|---|---|
| 풀 패킷 경로 분석(Hubble/Reachability 수준) | CNI/클라우드 API 연동 필요 → 앱 단독 불가 | **C1** 경로 스테퍼로 앱 관점 충분 |
| 진짜 GitOps desired-state diff(ArgoCD) | 선언적 config 소스(Git)가 우리에 없음 | **C4** 프로파일-기대로 대체 |
| 화면에서 env **직접 편집**(Kiali 인라인 수정) | env 는 K8s Secret/ConfigMap → 화면에서 못 고침(rollout 필요) | **C3** 검증까지만, 수정은 [런북](./통신디버깅-런북.md)·[레퍼런스](./쿠버배포-설정-레퍼런스.md) |
| AI 자동 원인추론(Komodor Klaudia) | 별도 모델/파이프라인, MVP 과함 | `fail_kind` 규칙 분류로 충분 |
| 요청/응답 **본문 캡처** | 클라이언트 transport 래핑 + 마스킹 비용 | 기존 [개선-추적표](./연동상태-통신디버깅-개선-추적표.md) B4-full 로 이관 |

## 5. 첫 구현 상세 (C1·C2 — 바로 가능)

근거: 둘 다 백엔드가 이미 반환하는 데이터만 씀.

**C1 연결 경로 스테퍼** — `Status.timing`(dns/connect/tls/server ms)와 `fail_kind`로 단계 통과 여부 도출:
- DNS 통과 = `timing.dns_ms>0` 또는 `resolved` 있음 / TCP = `connect_ms>0` / TLS = `tls` 있음 / 서버 = `reachable`.
- `fail_kind`가 `dns_fail`이면 DNS에서 ✗, `conn_refused`/`timeout`이면 TCP에서 ✗, `tls_fail`이면 TLS에서 ✗, `auth_fail`/`bad_status`면 서버에서 ✗.
- UI: 카드 `timing` 막대 위에 `DNS ✓ TCP ✓ TLS ✗ 서버 —` 스테퍼 한 줄.

**C2 영향 범위** — `required_by`(cap 배열) → [router.ts](../../web/src/router.ts) `capForPage` 역매핑 → 화면 이름:
- 상수맵 `CAP_LABEL = { dashboard:"관제", traces:"트레이스", guard:"가드레일", models:"모델", keys:"키·앱", users:"사용자", endpoints:"엔드포인트", ... }`.
- 실패 카드에 "이 의존성이 죽으면: 키·앱, 사용자 화면 비활성" 한 줄.
