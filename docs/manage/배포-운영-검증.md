# manage — 배포 · 운영 · 검증 런북

> `FABRIX_PROFILE=manage`(기본) 배포의 실무 절차. 각 단계에 **왜 그렇게 하는지 근거**를 붙였다. 아키텍처 배경: [아키텍처.md](아키텍처.md).

## 1. 배포 단위
| 구성 | 무엇 | 비고 |
|---|---|---|
| **web** | nginx + 정적 SPA | manage 는 전체 NAV |
| **backend(BFF)** | Go 단일 바이너리 | `FABRIX_PROFILE=manage`(기본, 미설정 시 자동) |
| (외부) 관측 + 데이터 스택 | Langfuse·ClickHouse·VM·게이트웨이·vLLM·Harbor·PostgreSQL | 읽기 + **쓰기 대상** |

## 2. 환경변수 (읽기 소스 + 쓰기 대상)
```bash
FABRIX_PROFILE=manage                  # 기본값(미설정도 manage)
# 읽기 (observe 와 동일) + 쓰기 대상
FABRIX_DATABASE_URL=postgres://fabrix:<pw>@fabrix-pg-rw.fabrix-endpoint:5432/fabrix  # 키/사용자/마스킹정책
FABRIX_KUBECTL=kubectl                  # 엔드포인트 CR apply/delete
FABRIX_ENDPOINTS_NS=dynamo-inference    # 엔드포인트 생성 기본 ns
FABRIX_SR_URL=http://semantic-router.vllm-semantic-router-system:8080  # 정책 변경·분류
FABRIX_HARBOR_URL=http://admin:<pw>@harbor-core.harbor                  # 모델 임포트
FABRIX_GEMMA_UPSTREAM=http://...:8000   # 플레이그라운드 테스트 호출
FABRIX_VMSELECT_URL=...  FABRIX_CLICKHOUSE_URL=...  FABRIX_LANGFUSE_*=...  # 읽기 소스
```
> **왜 PostgreSQL 이 manage 의 핵심인가** — 키·사용자·마스킹 정책의 **영속 저장소**다. 없으면 키 발급·RBAC·마스킹 편집이 비활성(503). observe 는 없어도 되지만 manage 는 사실상 필수.

## 3. RBAC (쓰기 권한 — 운영 보호 포함)
BFF ServiceAccount:
```
pods, pods/log: get, list, watch
DynamoGraphDeployment(CR): get, list, watch, create, delete       # 배포/삭제
secrets: get, create, update (자격증명 ns 한정)                    # HF/NGC 토큰
jobs: create (모델 임포트)
```
> **왜 create/delete 를 주되 라벨 가드를 두나** — manage 는 엔드포인트를 실제로 배포·삭제해야 하므로 write RBAC 가 필요하다. 그러나 코드가 **`fabrix.managed-by` 라벨이 붙은 CR 만 삭제**하도록 막아, RBAC 가 넓어도 고객의 다른 워크로드는 건드리지 못한다(권한 + 코드 가드의 이중 안전).
> **왜 secrets 권한을 ns 한정하나** — 자격증명(HF/NGC 토큰)은 특정 ns 의 Secret 에만 쓰면 된다. 클러스터 전역 secret 권한은 과도 → 최소권한.

## 4. 핵심 운영 절차
### 4.1 엔드포인트 배포
위저드(패턴·GPU·replica) → **preview**(`--dry-run=server` 서버검증) → **생성**(apply) → Pod 기동 확인(목록/로그).
> **왜 preview(dry-run) 를 먼저 거치나** — 잘못된 CR 로 GPU 노드를 점유·낭비하는 것을 막는다. 서버측 검증으로 스키마·쿼터 오류를 적용 전에 잡는다.

### 4.2 키 발급/회수
발급 시 평문 1회 표시(복사) → 이후 해시만. 회수는 `enabled=false`.
> **왜 1회만 평문인가** — 저장하지 않으므로(해시만) 재표시 불가. 운영자가 발급 즉시 안전한 곳에 보관하도록 강제 → 유출 표면 최소.

### 4.3 가드레일/마스킹 정책
- 가드레일(차단): 설정>가드레일>정책 — off/관찰(flag)/차단(block). **왜 관찰→차단 단계 전환** — 폐쇄망 첫 도입 시 오탐을 먼저 관찰로 확인하고 차단으로 올린다(서비스 중단 리스크 관리).
- 마스킹(관측 보존): 설정>가드레일>마스킹 — 캡처모드(none/masked/full) + PII 유형별 처리. 저장 시 PG 영속 → 게이트웨이 글루가 폴링 반영.

## 5. 검증 (배포 직후)
```bash
curl -s :8080/api/v1/capabilities | jq '.readonly, .profile'          # → false, "manage"
curl -s -o /dev/null -w '%{http_code}\n' -XPOST :8080/api/v1/endpoints # → 라우트 존재(400/로직)
curl -s :8080/api/v1/diagnostics | jq '.checks[] | {name, configured, reachable, error}'
# 엔드포인트 배포 위저드 E2E(preview→생성→목록 확인) — 실제 쓰기 경로 점검
```

## 6. 트러블슈팅
| 증상 | 원인 | 조치 |
|---|---|---|
| 키/마스킹 503 | PostgreSQL 미연결 | `FABRIX_DATABASE_URL`·PG 상태·`/diagnostics` postgresql |
| 엔드포인트 생성 실패 | kubectl/RBAC | SA 권한(create), `FABRIX_KUBECTL`, CR 스키마 |
| 정책 변경 503 | SR 미구성 | `FABRIX_SR_URL`·SR 상태 |
| 모델 임포트 실패 | Harbor/HF 토큰 | Harbor creds·자격증명 Secret |
| 마스킹 바꿔도 반영 안 됨 | 글루 폴링 주기/연결 | 글루 `FABRIX_BFF_URL`·`GLUE_MASKING_POLL_SECONDS` |

> 어느 경우든 1차 진단은 `GET /api/v1/diagnostics` — 어떤 의존성이 안 닿는지·왜인지(에러·지연)를 한 번에 본다.

## 7. NetworkPolicy
> **왜** — manage 는 쓰기 대상이 많다(k8s API·PG·SR·Harbor). 필요한 egress 만 열고 그 외는 막아 측면 이동·유출 경로를 좁힌다. Langfuse 셀프호스트면 외부 송신 없음(데이터 레지던시).
