# FABRIX Endpoint — 기능 구현 가능성 검증 (오픈소스 공식문서 대조)

목적: 대시보드의 모든 기능이 **실제 오픈소스로 구현 가능한지**를 공식 문서로 검증하고 확실히 한다.
방법: vLLM · NVIDIA Dynamo · VictoriaMetrics/VictoriaTraces · DCGM · ClickHouse · vLLM Semantic Router · MinIO · Harbor · Envoy · Langfuse 공식 docs/소스 직접 확인(2025–2026 현행).
판정: **✅ 가능**(공식 지원·이미 구현) · **🟡 조건부**(추가 컴포넌트/캡처 필요) · **🔴 불가/대체**(해당 도구로는 안 됨).

> 현재 백엔드 상태: `backend/internal/provider/live` 가 이미 Harbor·VictoriaMetrics(vmselect)·DCGM·ClickHouse·MinIO·Semantic Router·Dynamo CRD(kubectl)를 **실제 쿼리**한다. 트레이스만 `HasTraces:false`(집계 분해로 대체) 상태. 아래 표의 "현재"는 이 기준.

## 2. 조건부·주의 항목 상세 (실제 구현 시 필요한 것)

### 2-1. ★per-request 트레이스 — "단일 span + 속성", 별도 prefill/decode span 아님
- **사실**: vLLM은 `--otlp-traces-endpoint` 로 OTLP export 시 요청당 **`llm_request` span 1개**만 emit. prefill/decode/queue는 **그 span의 숫자 속성**(`gen_ai.latency.time_in_queue`, `time_in_model_prefill`, `time_in_model_decode`, `time_to_first_token`, `e2e`). **별도 child span이 아님.**
- **Dynamo는 분산 span을 emit**: `http-request`(root) → `kv_router.route_request/select_worker/schedule` → worker `handle_payload` (W3C trace context 전파, OTLP→Tempo). 즉 frontend/router/worker는 **진짜 span**, worker 내부 prefill/decode는 **속성**.
- **결론**: 우리 트레이스 waterfall은 **가능**. 단 데이터 출처는 (a) Dynamo 분산 span = proxy/router/queue 행, (b) vLLM `llm_request` span 속성 = prefill/decode "구간". 우리 뷰어가 prefill/decode를 별도 행으로 그리는 건 **시각화로는 유효하나, 실제론 worker span 속성을 구간 분해해 렌더**하는 것임. (DTO 주석에 반영 권장.)
- **필요**: vLLM에 `--otlp-traces-endpoint` (+ 상세 단계 원하면 `--collect-detailed-traces=all`, 성능비용 주의), Dynamo tracing 활성, VictoriaTraces OTLP 수집(`:10428/insert/opentelemetry/v1/traces`), 조회 `GET /select/jaeger/api/traces/{trace_id}`.
- 출처: docs.vllm.ai opentelemetry/design-metrics · docs.nvidia.com/dynamo tracing · docs.victoriametrics.com/victoriatraces.

### 2-2. Dynamo CRD 버전 (v1alpha1 → v1beta1)
- 현행 권장은 **`nvidia.com/v1beta1`**(`spec.components` 리스트, `resources.limits."nvidia.com/gpu"`). 우리 코드/생성기는 `v1alpha1`(`spec.services`, `resources.limits.gpu`) 사용 중 → **v1beta1 기준으로 생성기 갱신** 권장(operator가 round-trip 변환은 하지만 신규 문서/예제는 전부 v1beta1).

### 2-3. ★차단 프롬프트 원문 — Semantic Router는 원문을 보존하지 않음
- **사실**: Semantic Router는 PII 유형·jailbreak 여부·취한 액션(block/mask)을 **OTel span 속성/헤더(메타데이터)**로만 남기고, **입력 문장 원문 텍스트는 저장·노출하지 않음**. 헤더는 `x-vsr-selected-category/model/reasoning` 3종(라우팅 메타)뿐.
- **결론**: "무엇이 왜 걸렸나"(유형·사유)는 SR로 가능하지만, **"걸린 프롬프트 원문"은 별도 레이어에서 캡처해야 함.** 우리 구현(가드레일 상세 → Langfuse 원문 lazy 조회)이 **올바른 방향**. 단 전제:
  - Langfuse `observation.input` 에 원문이 들어가야 함 → **vLLM OTel 연동은 토큰/지연만 자동 전송, input/output 원문은 SDK로 명시 캡처해야 함**(자동 아님). 또는 ext_proc/프록시 단에서 request body 로깅.
  - 마스킹 정책을 걸면 원문이 가려짐(규제 대응 시 의도적).
- **필요**: 앱(또는 프록시)이 차단 케이스의 입력을 Langfuse GUARDRAIL observation `input`으로 기록하도록 계측. → 그러면 우리 화면의 "원문 불러오기"가 실데이터로 동작.
- 출처: github vllm-project/semantic-router · vllm-semantic-router.com (headers/observability) · langfuse.com/docs masking·observation-types.

### 2-4. ★엔드포인트 유휴 자동종료 — KEDA 필요 (Dynamo 단독 불가)
- **사실**: Dynamo Planner는 **min/max 사이 replica 자동조정(SLA 기반)**만 하고 **0으로 내리지 않음**. scale-to-zero는 **KEDA**(`ScaledObject`, Prometheus 트리거)로만. HPA는 0 불가.
- **결론**: 생성폼의 "유휴 N분 후 자동종료(→0)" 옵션은 **유효하나, 백엔드가 DGD와 함께 KEDA ScaledObject를 생성**해야 실현. (또는 옵션 라벨을 "유휴 시 최소 replica로 축소(Planner)"로 정직하게 조정.)
- **필요**: 클러스터에 KEDA 설치 + 생성 시 ScaledObject 동반 배포 로직.
- 출처: docs.nvidia.com/dynamo autoscaling(DGDSA/KEDA/HPA/Planner)·planner.

### 2-5. 쿼터 강제 — rpm은 표준, tpd(일일 토큰)는 토큰집계 필요
- **rpm(분당 요청)**: Envoy Gateway global ratelimit + Redis → 초과 시 **429**. ✅
- **tpd(일일 토큰 예산)**: **게이트웨이 단독 불가.** Envoy Gateway 1.3 "cost-based ratelimit"의 response cost를 **dynamic metadata(`ext_proc`/`token_count`)**에서 읽음 → **ext_proc(또는 자체 프록시)가 토큰을 집계해 metadata 주입**해야 함(unit=Day). 우리는 이미 ext_proc(Semantic Router) 경로가 있으니 여기에 토큰 미터링을 붙이거나 별도 미터링 프록시 필요. 🟡
- 출처: envoyproxy ratelimit · gateway.envoyproxy.io rate-limiting · Envoy Gateway 1.3 cost feature.

### 2-6. DCGM / MIG 효율
- 8개 DCGM 필드 전부 실재. **주의**: ① `DCGM_FI_PROF_SM_ACTIVE`는 dcgm-exporter 기본 CSV에서 **주석 처리(비활성)** → 활성화 필요. ② MIG 모드에선 레거시 `GPU_UTIL` 미지원 → `PROF_*`(per-MIG-instance, 라벨 `GPU_I_PROFILE`/`GPU_I_ID`) 사용. ③ "MIG 효율"은 인스턴스 프로파일 크기별 비선형 → **compute-slice 가중 정규화 + sanity check** 필요(단순 비율은 오해 소지).
- 출처: NVIDIA DCGM dcgm_fields.h · dcgm-exporter default-counters.csv · datacenter/dcgm gpu-telemetry.

### 2-7. Langfuse self-host 주의
- Sessions API·커스텀 모델 단가(`POST /api/public/models`)·observation input 저장 모두 self-host 가능. **단 Observations API v2 / Metrics API v2 가 현재 Cloud 우선일 수 있음** → self-host 버전에 맞는 엔드포인트(v1 fallback) 확인 필수. 비용은 Worker 서버측 계산(단가 등록 전제). 비용 단가는 ClickHouse 사용량 facet과 **단일 소스로 통일** 권장.
- 출처: langfuse.com/docs public-api·sessions·model-usage-and-cost·self-hosting.

### 2-8. Harbor 임포트
- Harbor REST(`/api/v2.0`)로 projects/repos/artifacts(pull_count·tags·size·push_time) 조회 ✅. skopeo copy는 **소스가 표준 OCI distribution이면 직행**, HF/NGC가 비OCI면 **OCI 아티팩트로 패키징 후 push** 단계 필요(🟡).

## 4. 종합 결론
- **전부 실제 구현 가능**하다. 막히는 항목(🔴)은 없고, **🟡 4건은 "추가 컴포넌트/계측이 전제"** 일 뿐 기술적으로 모두 실현 경로가 있다:
  1. 차단 프롬프트 원문 → 앱/프록시가 Langfuse `input` 캡처
  2. 유휴 자동종료 → KEDA ScaledObject 동반 배포
  3. 일일 토큰 예산 → ext_proc 토큰 미터링 + Envoy cost ratelimit
  4. per-request 트레이스 → vLLM OTLP + Dynamo tracing + VictoriaTraces 수집(또는 Langfuse)
- **이미 구현된(live) 부분**: GPU/MIG·트래픽·품질·사용량·가드레일 판정/증적·WORM·엔드포인트 CRUD·Harbor·플레이그라운드. 즉 핵심 관제는 이미 실데이터로 동작 가능.
- **성숙도 리스크**: VictoriaTraces 0.x(API 변경 가능) — PoC OK, 프로덕션은 버전 고정 또는 Langfuse 트레이스 병행.
