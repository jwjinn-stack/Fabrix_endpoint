## 0. 한 장 요약 (TL;DR)

- **계층 구조의 정석은 1996년 Shneiderman의 "Overview first, zoom and filter, details on demand"**(시각 정보 탐색 만트라)이다. 첫 화면은 요약, 상호작용으로 점점 구체화하라는 우리 의도와 정확히 일치한다. 모든 상용 관측 도구(Grafana/Datadog/Honeycomb)가 이 원칙 위에 서 있다.
- **drill-down(같은 데이터를 더 잘게)과 drill-through(다른 화면/데이터로 이동)는 다른 개념**이다. 우리는 둘 다 필요하다: 차트 안에서 차원을 쪼개는 drill-down + 패널→상세 대시보드로 점프하는 drill-through(deeplink).
- **고카디널리티 차원 분석의 레퍼런스 패턴은 Honeycomb BubbleUp**: 사용자가 이상 구간을 박스로 선택 → 시스템이 모든 차원을 자동 비교해 "어떤 차원값이 튀는지"를 히스토그램으로 보여준다. "groupby 느낌으로 자동 해석"의 정확한 사례.
- **대시보드의 AI 해석 표준은 MCP(Model Context Protocol)**. Grafana는 공식 MCP 서버를 제공하며, **serviceAccount 토큰 + RBAC**로 인증하고, 큰 대시보드 JSON 대신 **`get_dashboard_summary`/`get_dashboard_panel_queries`** 같은 토큰 절약형 도구를 노출한다. Datadog도 MCP 서버 + Bits AI로 자연어→위젯/대시보드 생성을 한다.
- **세 데이터 소스를 묶는 공통 차원**: `model` · `endpoint` · `namespace`(=Dynamo `dynamo_namespace`) · `time window` · `status` · (Langfuse) `user`/`session`. 이 공통 라벨 축이 "groupby 해석"의 뼈대가 된다.
- **우리 적용안**: ① 각 path를 overview→drill 계층으로 재배치, ② 세 소스를 공통 차원 모델로 정규화, ③ 대시보드 메타데이터(패널·쿼리·차원·의미)를 MCP resource/tool로 노출하는 자체 MCP 서버(FABRIX endpoint MCP). 상세는 §5.

## 1. 계층적·점진적 공개(Progressive Disclosure) 대시보드 패턴

### 1.1 이론적 토대 — Shneiderman 만트라
- Ben Shneiderman, *"The Eyes Have It: A Task by Data Type Taxonomy for Information Visualizations"* (1996)의 핵심 명제:
  **"Overview first, zoom and filter, then details-on-demand."**
  1. **Overview first** — 사용자가 가장 먼저 보는 요약. 전체 그림을 먼저 보여주고 탐색을 유도.
  2. **Zoom & Filter** — 날짜 범위 선택, 차원 필터, 인터랙티브 범례로 관심 영역만 좁힘.
  3. **Details on demand** — 개별 포인트 클릭 시 상세 정보 제공.
- 이 만트라는 정보 시각화/대시보드 설계의 사실상 표준 프레임이며 산업 전반에 채택됨.
- 참고: [InfoVis Wiki — Visual Information-Seeking Mantra](https://infovis-wiki.net/wiki/Visual_Information-Seeking_Mantra), [Recorded Future — Information-Seeking Mantra in Dashboards](https://www.recordedfuture.com/blog/information-seeking-mantra), [Georgia Tech 강의자료(PDF)](https://faculty.cc.gatech.edu/~john.stasko/8001/craft05.pdf)

### 1.2 Drill-down vs Drill-through (개념 구분이 중요)
- **Drill-down**: 같은 맥락 안에서 데이터를 더 잘게 — 예: 전체 latency → model별 → endpoint별 → 개별 request. (차원 분해/groupby 심화)
- **Drill-through**: 한 패널에서 *다른* 대시보드/도구/원본 시스템으로 점프 — 예: latency 패널 → 해당 모델의 trace 상세 대시보드, 로그 쿼리, Langfuse trace.
- Grafana 공식 권장: "계층형 대시보드 + 다음 레벨로의 drill-down"을 설계 패턴으로 제시. 각 시각화에서 상세 대시보드/로그쿼리/디버깅 도구/시스템 자체로 **링크**하라. 값 범위가 너무 크면 전부 말고 **Top-N 문제군**만 보여줄 것.
- 참고: [Grafana — Dashboard best practices](https://grafana.com/docs/grafana/latest/visualizations/dashboards/build-dashboards/best-practices/), [Grafana Drilldown](https://grafana.com/docs/grafana/latest/visualizations/simplified-exploration/), [Grafana Metrics Drilldown](https://grafana.com/docs/grafana/latest/visualizations/simplified-exploration/metrics/drill-down-metrics/)

### 1.3 고카디널리티 drill-down의 레퍼런스 — Honeycomb BubbleUp
- **동작**: 사용자가 heatmap 위에서 이상 구간을 박스로 드래그 → BubbleUp이 **모든 차원(필드)을 한꺼번에** 비교하여, 선택 구간 vs 베이스라인(나머지 이벤트)의 분포 차이를 히스토그램으로 자동 정렬. "어떤 차원값이 이 문제를 일으키는가"를 사람이 일일이 groupby 하지 않아도 찾아준다.
- **의의**: 카디널리티가 수백만이어도(개별 user/request 단위) 증상→정확한 outlier 차원까지 수 분 내 도달. 집계로는 정상처럼 보이는 "특정 고객만 느림"을 잡아냄.
- 이것이 우리가 말한 **"groupby 느낌으로 자동 해석"**의 가장 성숙한 제품 사례.
- 참고: [Honeycomb — High Cardinality](https://docs.honeycomb.io/get-started/observability/concepts/high-cardinality), [Honeycomb — Understanding High Cardinality](https://www.honeycomb.io/resources/getting-started/understanding-high-cardinality-role-observability), [BubbleUp 사례 블로그](https://blog.pvincent.io/2021/06/bubbling-up-customer-specific-experience-with-high-cardinality-observability/)

### 1.4 제품별 계층 구성 요약
| 제품 | overview 층 | drill 메커니즘 | 비고 |
|---|---|---|---|
| **Grafana** | 폴더/요약 대시보드 | Data Links·변수·Drilldown 앱·deeplink로 상세 대시보드 이동 | 계층형 대시보드를 공식 권장 |
| **Datadog** | 요약 대시보드 | 위젯 클릭→관련 대시보드/로그/APM, "자주 파고드는 정보"를 상위에 | 자연어→위젯 생성(Bits) |
| **Honeycomb** | heatmap 분포 | BubbleUp 박스 선택→전 차원 자동 비교 | 고카디널리티 강점 |

## 2. 대시보드를 AI(LLM)가 해석하게 만들기 — MCP 중심

### 2.1 Grafana MCP Server (= 우리가 말한 "serviceAccount로 대시보드 해석"의 표준)
- Grafana 공식 MCP 서버([github.com/grafana/mcp-grafana](https://github.com/grafana/mcp-grafana))는 Claude Desktop·Cursor·VS Code Copilot 등 MCP 클라이언트가 Grafana를 읽/쓰게 해준다.
- **인증**: Grafana **serviceAccount 토큰**(또는 id/pw)으로 Grafana API 호출. 각 도구는 Grafana RBAC 권한·스코프를 요구(예: 대시보드 읽기 = `dashboards:read` + `dashboards:uid:xyz`).
- **읽기 전용 모드**: `--disable-write`로 쓰기 차단 → 권한 최소화된 serviceAccount로 안전하게 관측 데이터만 노출. (우리 **observe(삼성증권 read-only)** 프로파일과 정확히 같은 사고방식)
- **대시보드 해석에 쓰이는 핵심 도구**(토큰 절약형):
  - `search_dashboards` — 대시보드 검색
  - `get_dashboard_summary` — **풀 JSON 대신 압축 요약** (LLM이 먼저 이걸 본다)
  - `get_dashboard_property` — JSONPath로 특정 섹션만 추출
  - `get_dashboard_panel_queries` — **패널 제목·쿼리·datasource·타입** 추출 (대시보드의 "의미"를 LLM에 전달하는 핵심)
  - `get_panel_image` — 패널을 PNG 렌더 (멀티모달 해석용)
  - `generate_deeplink` — Grafana 리소스로의 정확한 딥링크 생성
- **데이터 질의 도구**(Prometheus 예): `query_prometheus`(PromQL), `list_prometheus_metric_names`, `list_prometheus_metric_metadata`, `list_prometheus_label_names`, `list_prometheus_label_values`, `query_prometheus_histogram`(분위수). → **라벨 이름/값 나열 도구가 곧 "groupby 가능한 차원"을 LLM에 알려주는 장치**.
- **컨텍스트 윈도우 전략**: 큰 대시보드 JSON을 그대로 주지 말고 summary/property 우선. 일부 도구는 기본 비활성(`run_panel_query`, admin류).
- 참고: [Grafana MCP 문서](https://grafana.com/docs/grafana/latest/developer-resources/mcp/), [Introduction](https://grafana.com/docs/grafana/latest/developer-resources/mcp/introduction/), [MCP tools reference](https://grafana.com/docs/grafana/latest/developer-resources/mcp/reference/mcp-tools-table/), [app plugin에서 LLM+MCP 사용](https://grafana.com/developers/plugin-tools/how-to-guides/app-plugins/use-llms-and-mcp)

### 2.2 Datadog — MCP Server + Bits AI + Dashboard Agent
- **Datadog MCP Server**: 관측 데이터와 AI 에이전트(Cursor/Codex/Claude Code/자체 에이전트) 사이 브리지. trace를 ML앱·에러상태·태그로 검색, span 트리 시각화, 에이전트 루프 조사 등.
- **Bits AI / Bits Assistant**: 웹·모바일·Slack에서 자연어로 검색·시각화·액션을 한 번에.
- **Dashboard Agent 아키텍처(중요한 구현 교훈)**:
  - **Widget Agent**: "프로덕션 호스트의 평균 CPU를 시계열로 보여줘" → 유효한 위젯 JSON 생성.
  - **Dashboard Agent**: 요청을 위젯 단위로 분해 → 각 위젯 생성 → **메트릭 메타데이터를 가져오는 tool 호출**을 여러 번 오케스트레이션.
  - **스키마로 grounding**: OpenAI structured output로 위젯/대시보드 JSON 스키마를 강제 → 프로그램적 검증 용이, 후처리 불필요.
  - **평가**: 결정적 스키마 검증 + LLM-as-judge(이진 점수)로 "의도 대비 의미적 정확도" 측정(정확한 JSON 일치를 요구하지 않음).
  - **교훈**: 관측 없이는 "malformed JSON/불완전 대시보드"가 조용한 degradation으로 나타남 → 어디서 실패했는지(프롬프트/모델/인프라) trace로 드러내야 함.
- 참고: [Datadog MCP Server](https://docs.datadoghq.com/bits_ai/mcp_server/), [LLM Obs MCP](https://docs.datadoghq.com/llm_observability/mcp_server/), [Building dashboard agents 블로그](https://www.datadoghq.com/blog/llm-observability-at-datadog-dashboards/), [Bits Assistant 소개](https://www.datadoghq.com/blog/introducing-bits-assistant/)

### 2.3 MCP 서버 설계 원칙 (관측·LLM 친화)
- **도메인 동사로 도구를 노출**하라. `create_record`/`update_row` 같은 범용 메서드 대신 `submit_expense_report`처럼 의미가 분명한 동작. → 관측 맥락에선 `summarize_endpoint_health`, `compare_models_by_latency`, `top_cost_drivers` 같은 **인사이트 동사**.
- **시맨틱 메타데이터 필수**: name·description·입출력 스키마·비용·권한 레벨. "세 살 아이도 이해할 도구"가 좋은 도구.
- **OpenTelemetry GenAI 시맨틱 컨벤션** 준수: `gen_ai.tool.name`, `mcp.server.name` 등 속성.
- **3계층 관측**: Transport/Protocol(핸드셰이크·세션·JSON-RPC 에러코드), Tool Execution, Agentic Performance.
- **구조화 로그**: trace id/span id 포함, 세션·유저·서버·도구·에러타입별 집계 가능, 민감정보 배제.
- 참고: [Designing MCP Servers for Observability](https://blog.dreamfactory.com/designing-mcp-servers-for-observability), [MCP Best Practices](https://oshea00.github.io/posts/mcp-practices/), [MCP Semantic Layer](https://colrows.com/blogs/mcp-semantic-layer-integration/), [Alibaba — MCP for Observability 2.0](https://www.alibabacloud.com/blog/mcp-for-observability-2-0---six-practices-for-making-good-use-of-mcp_602423)

## 3. 관측 시맨틱 — 세 소스가 노출하는 것 & groupby 차원

### 3.1 NVIDIA Dynamo (추론 서빙)
- 메트릭 노출: HTTP Frontend가 **port 8000**(`--http-port`/`DYN_HTTP_PORT`)의 `/metrics`에 Prometheus 포맷으로. (백엔드 워커는 별도 포트, 예전엔 6881 언급)
- 접두사: 모든 Dynamo 메트릭 `dynamo_*`, 프론트엔드 `dynamo_frontend_*`.
- **공통 라벨(=groupby 차원)**: `dynamo_namespace`, `dynamo_component`, `dynamo_endpoint`, 그리고 frontend 메트릭의 **`model`**.
- 핵심 메트릭:
  - `dynamo_frontend_requests_total` (counter) — 총 요청
  - `dynamo_frontend_inflight_requests` / `dynamo_frontend_queued_requests` (gauge) — 동시처리/큐 대기
  - `dynamo_frontend_time_to_first_token_seconds` (histogram) — **TTFT**
  - `dynamo_frontend_inter_token_latency_seconds` (histogram) — **ITL**
  - `dynamo_frontend_request_duration_seconds` (histogram) — 요청 전체 지연
  - `dynamo_frontend_input_sequence_tokens` / `output_sequence_tokens` (histogram) — 입·출력 토큰 길이
  - 모델 설정: `dynamo_frontend_model_context_length`, `..._kv_cache_block_size`, `..._max_num_batched_tokens` 등
- **disaggregated(분리형) 서빙**: prefill(프롬프트 처리)와 decode(토큰 생성)를 별도 GPU 풀로 분리·독립 스케일. KV cache 전송시간·큐 대기·처리시간이 Planner 의사결정의 입력. Planner는 GPU 용량 지표 + SLO(TTFT, ITL)로 라우팅.
- 백엔드별(vLLM/TRT-LLM/SGLang) 메트릭이 통일 추상화로 KV-aware 라우팅·Prometheus parity·OTel 트레이싱을 공유.
- 참고: [Dynamo Prometheus/Grafana 시각화 문서](https://docs.nvidia.com/dynamo/archive/0.6.1/observability/prometheus-grafana.html), [Dynamo 소개 블로그](https://developer.nvidia.com/blog/introducing-nvidia-dynamo-a-low-latency-distributed-inference-framework-for-scaling-reasoning-ai-models/), [KV Cache 병목 블로그](https://developer.nvidia.com/blog/how-to-reduce-kv-cache-bottlenecks-with-nvidia-dynamo/), [GitHub ai-dynamo/dynamo](https://github.com/ai-dynamo/dynamo)

> 참고(백엔드 vLLM 메트릭, Dynamo가 위로 노출/매핑): `vllm:num_requests_running|waiting|swapped`, `vllm:kv_cache_usage_perc`, `vllm:time_to_first_token_seconds`, `vllm:inter_token_latency_seconds`, `vllm:e2e_request_latency_seconds`, `vllm:request_prefill_time_seconds`, `vllm:request_decode_time_seconds`. ([vLLM Metrics 문서](https://docs.vllm.ai/en/latest/design/metrics/))

### 3.2 Langfuse (LLM 관측·평가)
- **Metrics API**(커스텀 분석)의 질의 모델:
  - **View**: `traces`, `observations`(v2 권장, 더 빠르고 강력), `scores-numeric`, `scores-categorical`.
  - **Measure**: `count`, `latency`, `totalCost`, `totalTokens`, `timeToFirstToken`.
  - **Aggregation**: `sum`/`avg`/`count`/`max`/`min`/`p50`/`p75`/`p90`/`p95`/`p99`.
  - **groupby 차원**:
    - trace: `name`, `tags`, `release`, `version`, `environment`, `observationName`, `scoreName`
    - observation: `type`, `name`, `level`, `version`, **`providedModelName`**, `promptName`, `promptVersion`, `traceName`, `traceRelease`, `traceVersion`, `scoreName`
    - score: `name`, `environment`, `source`, `dataType`, `observationModelName`, `observationPromptName`, `stringValue`(categorical) 등
  - **시간 입자**: `hour`/`day`/`week`/`month`/`auto`, `fromTimestamp`·`toTimestamp` 필수.
  - **고카디널리티 주의**: `id`, `traceId`, `userId`, `sessionId`는 **groupby 불가**(필터로만 사용) — 성능 때문. (Honeycomb과 달리 Langfuse 집계는 고카디널리티 그룹핑을 막음 → 우리 BubbleUp형 분석은 메트릭이 아니라 trace 탐색/필터로 풀어야 함)
- **기본 대시보드**: Latency / Cost / Usage. cost·latency를 user·session·geography·feature·model·prompt version으로 분해.
- 참고: [Langfuse Metrics API](https://langfuse.com/docs/metrics/features/metrics-api), [Metrics Overview](https://langfuse.com/docs/metrics/overview), [v2 Metrics & Observations API](https://langfuse.com/changelog/2025-12-17-v2-metrics-and-observations-api), [Custom Dashboards](https://langfuse.com/docs/metrics/features/custom-dashboards), [Observations API](https://langfuse.com/docs/api-and-data-platform/features/observations-api). (사내: [langfuse-api.md](../integration/langfuse-api.md), [langfuse-mcp.md](../integration/langfuse-mcp.md), [langfuse-기능정리.md](../langfuse-기능정리.md))

### 3.3 Prometheus / VictoriaMetrics (메트릭 모델)
- 데이터 모델 = `metric_name{label1="v1", label2="v2"}` 시계열. **라벨 = groupby 축**. PromQL `sum by (model, endpoint)(...)` 형태로 차원 집계.
- 분위수는 histogram + `histogram_quantile`. (Grafana MCP의 `query_prometheus_histogram`이 이를 추상화)
- 우리는 VictoriaMetrics 사용([metrics-victoriametrics.md](../integration/metrics-victoriametrics.md)) — PromQL/MetricsQL 호환.

### 3.4 세 소스를 묶는 공통 차원 모델 (★ groupby의 뼈대)
| 공통 차원 | Dynamo 라벨 | Langfuse 차원 | Prometheus/VM 라벨 |
|---|---|---|---|
| **model** | `model` | `providedModelName`/`observationModelName` | `model` |
| **endpoint** | `dynamo_endpoint` | `traceName`/`name` | `endpoint`/`handler` |
| **namespace/tenant** | `dynamo_namespace` | `environment`/`tags` | `namespace` |
| **status** | (request 결과) | observation `level` | `status`/`code` |
| **time window** | scrape ts | `timeDimension`(hour/day/…) | range vector |
| **user/session** | — | `userId`/`sessionId`(필터 전용) | (보통 없음) |
| **cost/tokens** | input/output token hist | `totalCost`/`totalTokens` | — |
| **latency** | TTFT/ITL/duration hist | `latency`/`timeToFirstToken` | `*_seconds` hist |

→ **`model` × `endpoint` × `namespace` × `time` 가 4대 공통 축.** 이 위에서 "이 모델의 이 엔드포인트가 이 시간대에 TTFT p95가 튀고(=Dynamo) 그 구간 trace 비용이 급증(=Langfuse)" 식의 cross-source 해석이 가능해진다.

## 4. "AI가 대시보드를 읽고 groupby로 해석"하는 실제 패턴 정리

1. **Grafana 패턴(읽기 해석형)**: serviceAccount(read-only) → MCP → `get_dashboard_summary`로 대시보드 의미 파악 → `list_*_label_values`로 가능한 groupby 차원 확인 → `query_prometheus(_histogram)`로 분위수/차원 집계 → 자연어 인사이트. 우리가 따라갈 1순위 레퍼런스.
2. **Datadog 패턴(생성형)**: 자연어 → 위젯/대시보드 JSON 생성(스키마 강제) + 메타데이터 tool 호출. "대시보드를 만들어주는" 방향.
3. **Honeycomb 패턴(자동 차원분석)**: 사람이 이상구간 선택 → 전 차원 자동 비교(BubbleUp). LLM 없이도 groupby 자동화. 우리의 "다른 action → 더 구체" 의 핵심 영감.

## 5. FABRIX endpoint 적용 권장안

### 5.1 대시보드를 3계층으로 재배치 (Overview → Group → Detail)
- **L1 Overview (path 첫 화면)**: 추상 요약만. 예) 엔드포인트별 건강도 카드(요청량·TTFT p95·에러율·비용), 이상 Top-N. 클릭 유도가 목적이지 분석이 목적이 아님.
- **L2 Group (1차 action = drill-down)**: 공통 차원으로 groupby. 예) `model`별 / `endpoint`별 / `namespace`별 latency·cost·throughput 분해. 여기서 "어느 그룹이 튀는지"를 본다 (BubbleUp 영감: 자동으로 outlier 그룹 강조).
- **L3 Detail (2차 action = drill-through)**: 개별 request/trace로 점프. Dynamo 메트릭 구간 → 해당 시간·모델의 Langfuse trace 상세(span/generation/cost/score)로 deeplink.
- 구현 힌트: 패널마다 **data link/deeplink**를 부여(Grafana 권장). L2→L3는 같은 4대 공통 축(model/endpoint/namespace/time)으로 컨텍스트를 전달.

### 5.2 공통 차원 정규화 레이어 (백엔드 Go)
- Dynamo(VictoriaMetrics) · Langfuse Metrics API · 메트릭을 **단일 차원 스키마**(model/endpoint/namespace/status/time/user)로 정규화하는 어댑터를 둔다.
- 각 패널 응답에 **"이 패널이 보여주는 것 + groupby 가능 차원 + 단위/의미"** 메타데이터를 함께 실어 보낸다(프론트 drill-down UI와 MCP 양쪽이 재사용).

### 5.3 FABRIX endpoint MCP 서버 (대시보드 AI 해석)
Grafana MCP를 레퍼런스로, 우리 대시보드 의미를 노출하는 자체 MCP 서버:
- **인증**: 프로파일 기반 토큰. **observe(삼성증권)=read-only**(`--disable-write` 사고방식), manage=풀. [fabrix-two-profile-deploy] 와 정합.
- **resource**: 대시보드 카탈로그(path·패널·차원·단위·소스). LLM이 "어떤 화면에 무엇이 있는지"를 먼저 읽는 용도.
- **인사이트 동사형 tool**(범용 쿼리 X):
  - `summarize_endpoint_health(endpoint?, window)` → 핵심 KPI 요약
  - `groupby_metric(measure, by=[model|endpoint|namespace], window, agg=p95|avg|sum)` → 차원 분해
  - `top_outliers(measure, by, window)` → BubbleUp형 이상 그룹 (Top-N)
  - `explain_panel(path, panel_id)` → 패널 의미·현재값·이상여부 자연어 설명 (Grafana `get_dashboard_summary` 대응)
  - `trace_drilldown(model, endpoint, time_range)` → Langfuse trace 상세로 연결 (L3)
  - `deeplink(path, filters)` → 프론트 화면 딥링크 생성
- **컨텍스트 절약**: 풀 데이터 대신 요약/Top-N 우선, 큰 payload는 deeplink로. (Grafana 교훈)
- **스키마 강제 + 관측**: 응답 JSON 스키마 고정, MCP 호출 자체를 trace(어느 도구·차원·실패지점). (Datadog 교훈)
- 기존 [langfuse-mcp.md](../integration/langfuse-mcp.md) 와 통합/정합 검토 필요 — Langfuse MCP는 trace 소스로, FABRIX MCP는 대시보드 의미·cross-source groupby 레이어로 역할 분리.

### 5.4 단계적 도입 로드맵
1. **공통 차원 스키마 확정**(§3.4) + 패널 메타데이터 부착 — drill-down UI·MCP 공통 기반.
2. **L1/L2/L3 계층 재배치** — 프론트 drill-down/deeplink부터 (백엔드 변경 최소).
3. **`groupby_metric`/`top_outliers` 백엔드 API** — BubbleUp형 자동 그룹 강조.
4. **FABRIX MCP 서버** — read-only(observe) 우선, 인사이트 동사 tool + 대시보드 resource.
5. **(선택) 생성형** — 자연어→패널 구성(Datadog식)은 마지막.

# 심화 리서치 Round 1 — "메트릭을 무엇에 쓰는가" (활용 중심)

> 파생 아이디어: §3은 "메트릭이 무엇인가"였다. 운영 인사이트는 메트릭을 **의사결정(스케일·라우팅·비용·품질)** 으로 바꿀 때 생긴다. 그리고 메트릭↔trace를 잇는 **기술적 다리**가 있어야 L2→L3 drill-through가 실현된다.

## R1-1. Dynamo 메트릭의 활용 — SLA Planner (TTFT/ITL → 오토스케일)
- **SLA-based Planner**: TTFT·ITL **목표(SLO)** 를 정하면 prefill/decode 워커 수를 자동 조정. 입력 신호 = frontend `/metrics`의 요청수·ISL(input seq len)·OSL(output seq len)·TTFT·ITL + **prefill queue 길이**, **decode memory 사용량**.
- **사전 프로파일링 + Interpolator**: 배포 전 모델별 ISL↔TTFT/prefill throughput를 프로파일 → Planner가 "기대 TTFT/ITL"을 추정·보정. 실제값이 이상값과 다르면 calibrate.
- **핵심 통찰(대시보드 해석에 직결)**: *TTFT는 queueing과 prefix cache hit rate에 강하게 의존*. 요청이 한꺼번에 몰리면 큐 적체로 TTFT 급등, prefix cache hit이 높으면 실제 prefill 토큰이 적어 TTFT 급락. → **TTFT 이상치를 볼 때는 항상 queue·cache hit을 같이 봐야 함**(단일 지표 오독 방지). 이게 "groupby + 맥락" 해석의 실제 예.
- 활용: TTFT p95를 (CPU/GPU util 대신) **오토스케일 신호**로 사용.
- 참고: [Dynamo 0.4 — SLO 오토스케일·실시간 관측](https://developer.nvidia.com/blog/dynamo-0-4-delivers-4x-faster-performance-slo-based-autoscaling-and-real-time-observability/), [SLA-based Planner 문서](https://docs.nvidia.com/dynamo/latest/architecture/sla_planner.html), [Planner Design](https://docs.nvidia.com/dynamo/design-docs/component-design/planner-design), [sla_planner.md](https://github.com/ai-dynamo/dynamo/blob/main/docs/planner/sla_planner.md)

## R1-2. KV/prefix cache hit rate — 비용·TTFT의 숨은 변수
- **핵심 지표**: `vllm:gpu_prefix_cache_hit_rate` (구버전 `vllm:gpu_prefix_cache_queries_total`/`_hits_total`). Dynamo가 위로 노출.
- **효과(실측 사례)**: warm cache hit 시 TTFT 88% 단축, hit rate 87%. prefix-aware 라우팅으로 P90 TTFT 92s→0.542s, throughput 2배.
- **운영 임계치(알림 룰 후보)**: cache 사용률 > 90%, **hit rate < 50%**, 높은 eviction rate → 메모리 증설/프롬프트 최적화 신호.
- **라우팅 trade-off**: 같은 system prompt 80%면 prefix-aware가 한 GPU로 몰아줌 → in-flight가 중앙값 2배 초과 시 load-aware fallback으로 분산. (대시보드에서 cache hit ↑ 인데 특정 GPU만 과부하 = 이 trade-off를 봐야)
- 활용: **"비용 대시보드"의 1차 드라이버 = cache hit rate**. 비용 이상 → cache hit/eviction으로 drill-down.
- 참고: [KV Cache Locality (Ranvier)](https://ranvier.systems/2026/04/30/kv-cache-locality-the-hidden-variable-in-your-llm-serving-cost.html), [Red Hat — llm-d KV-aware 라우팅](https://developers.redhat.com/articles/2025/10/07/master-kv-cache-aware-routing-llm-d-efficient-ai-inference), [DigitalOcean — prefix-aware routing](https://www.digitalocean.com/blog/reduce-llm-inference-costs-prefix-caching), [Ray prefix-aware routing](https://docs.ray.io/en/latest/serve/llm/user-guides/prefix-aware-routing.html)

## R1-3. Prefill/Decode 분리 — 병목 진단 매트릭스
- **두 가지 병목**: ① **Prefill 병목** = prefill 큐 적체 + decode replica 유휴, ② **Decode 병목** = prefill은 빠른데 decode가 못 따라감. + **HOL(Head-of-Line) blocking**: 긴 prefill이 GPU 점유 → decode 요청이 뒤에서 대기.
- **진단 지표**: TTFT·**TPOT**(Time Per Output Token)·ITL·**KV cache 전송 지연**·GPU util·**두 개의 분리된 큐**(prefill queue / decode queue).
- **스케일 휴리스틱**: decode 큐 > N → decode replica 증설, prefill 큐 > M → prefill replica 증설(독립). decode TPS가 협조적 스케일링의 견고한 지표.
- 활용: 대시보드 L2에서 **"병목 위치"를 단일 카드로** — (prefill큐, decode큐, KV전송, GPU util) 조합으로 ①/② 자동 판정. (BubbleUp형 "어디가 튀나"의 도메인 특화 버전)
- 참고: [Ray — P/D disaggregation](https://docs.ray.io/en/latest/serve/llm/architecture/serving-patterns/prefill-decode.html), [Nexus(arXiv) — intra-GPU 분리](https://arxiv.org/html/2507.06608v5), [Spheron — P/D 2x throughput 가이드](https://www.spheron.network/blog/prefill-decode-disaggregation-gpu-cloud/)

## R1-4. Langfuse trace의 활용 — score를 알림 대상으로
- **Score = 만능 평가 객체**: name·value·dataType(NUMERIC/CATEGORICAL/BOOLEAN/TEXT). 출처 = 휴먼 어노테이션 / **LLM-as-judge** / 코드 평가 / 엔드유저 피드백 — 모두 score로 통일 저장.
- **온라인 vs 오프라인**: 온라인 = 프로덕션 트래픽 **샘플**에 실시간 평가(drift 조기 탐지). 오프라인 = 배포 전 dataset 실험(CI에서 regression이면 deploy 차단).
- **핵심 운영 원칙**: *"score를 알림 대상 메트릭으로 다뤄라 — faithfulness score 급락이 회귀의 최초 경고"*. (지연·비용뿐 아니라 **품질**도 시계열·알림 대상)
- 활용: 품질 대시보드 = score를 model/prompt version별 groupby + 시계열. 급락 시 해당 trace로 drill-through.
- 참고: [Langfuse Evaluation 개념](https://langfuse.com/docs/evaluation/core-concepts), [LLM-as-a-Judge](https://langfuse.com/docs/evaluation/evaluation-methods/llm-as-a-judge), [Automated Evaluations](https://langfuse.com/blog/2025-09-05-automated-evaluations)

## R1-5. ★ 메트릭↔Trace를 잇는 다리 — OTel GenAI 컨벤션 + Prometheus Exemplars
이게 Round 1의 가장 중요한 발견. **L2(메트릭) → L3(trace) drill-through의 실제 메커니즘.**
- **OTel GenAI Semantic Conventions** = 벤더 중립 공통 어휘. 핵심 속성: `gen_ai.request.model`, `gen_ai.usage.input_tokens`/`output_tokens`, `gen_ai.response.finish_reasons`, 그리고 `gen_ai.tool.name`/`mcp.server.name`(§2.3). → **세 소스를 한 vocabulary로 묶는 표준 접착제.** Datadog 등 주요 벤더가 네이티브 지원. (2026.03 기준 다수 experimental, `OTEL_SEMCONV_STABILITY_OPT_IN`로 dual-emit)
- **Prometheus Exemplars**: 히스토그램/카운터 샘플에 **trace id를 첨부**. Grafana에서 latency 히스토그램의 **p99 점을 클릭 → 그 지연을 만든 정확한 trace로 점프**. OpenMetrics 포맷으로 노출.
- → **우리 설계 결론**: Dynamo의 `*_seconds` 히스토그램(TTFT/ITL/duration)에 **exemplar로 trace id**를 실으면, "TTFT p99가 튄 구간 → 그 요청의 Langfuse trace(비용·프롬프트·score)"로 한 번에 내려갈 수 있다. **L2→L3 drill-through = exemplar로 구현.** 공통 라벨(model/endpoint/namespace)이 exemplar에도 실려야 함.
- 참고: [OTel — GenAI Observability](https://opentelemetry.io/blog/2026/genai-observability/), [OTel GenAI spans spec](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/), [Datadog — OTel GenAI 지원](https://www.datadoghq.com/blog/llm-otel-semantic-convention/), [Exemplars로 metrics→traces 점프](https://vbehar.medium.com/using-prometheus-exemplars-to-jump-from-metrics-to-traces-in-grafana-249e721d4192), [Grafana Metrics Drilldown(exemplar 통합)](https://github.com/grafana/metrics-drilldown), [Tempo — metrics from traces](https://grafana.com/docs/tempo/latest/metrics-from-traces/)

# 심화 리서치 Round 2 — 전구간 상관 · 계층화 방법론 · 비용귀속 · AI 해석의 grounding

> 파생 아이디어: R1-5에서 "exemplar로 metric→trace"가 가능함을 봤다. 그러려면 ① **세 소스가 같은 trace id를 공유**해야 하고(전구간 propagation + Langfuse OTel 수집), ② 대시보드를 **검증된 방법론(Golden/RED/USE)** 으로 계층화해야 하며, ③ AI 해석이 신뢰성을 가지려면 **시맨틱 레이어**가 필요하다.

## R2-1. ★ Langfuse = OTel 백엔드 → 세 소스를 한 trace로 수집
- Langfuse는 **OpenTelemetry 백엔드**로 동작: OTLP 수신 엔드포인트 **`/api/public/otel`**, Basic Auth(base64 API key). OTel Collector에서 `otlphttp/langfuse` exporter로 export 가능.
- 진화 중인 GenAI semantic conventions를 Langfuse 데이터 모델로 **매핑**해서 수신. OpenLLMetry/OpenLIT 라이브러리로 Java·**Go** 등 언어/프레임워크 확장. **vLLM→Langfuse(OTel)** 통합 가이드 존재.
- → **설계 함의**: 우리 Go 게이트웨이/Dynamo에서 OTLP로 span을 내보내면 Langfuse가 trace로 수집. 즉 *별도 SDK 없이 OTel 한 줄기로* 메트릭(Prometheus/VM)·trace(Langfuse) 양쪽을 채울 수 있고, **trace id가 공유**된다.
- 참고: [Langfuse — OTel 통합](https://langfuse.com/integrations/native/opentelemetry), [기존 OTel 셋업과 통합 FAQ](https://langfuse.com/faq/all/existing-otel-setup), [vLLM→Langfuse(OTel)](https://langfuse.com/integrations/model-providers/vllm)

## R2-2. W3C Trace Context — 단일 trace id를 전구간에 전파
- **traceparent**(핵심 trace 신원) + **tracestate**(벤더 컨텍스트) 헤더로 벤더 중립 전파. **게이트웨이가 인바운드 요청에 traceparent 생성**(edge-first) → 모든 컴포넌트가 forward.
- **함정**: *한 서비스라도 헤더를 forward 안 하면 trace가 끊긴 조각들로 분해*되어 추적 무의미. → 게이트웨이→Dynamo→백엔드 전 구간에서 헤더 보존이 필수 요구사항.
- → **설계 함의**: 우리 게이트웨이(Go)가 traceparent를 만들고, Dynamo 메트릭 **exemplar의 trace id == Langfuse trace id == 게이트웨이 trace id** 가 되도록 맞춘다. 이게 L2→L3 drill-through의 전제.
- 참고: [W3C Trace Context(Dynatrace)](https://www.dynatrace.com/knowledge-base/w3c-trace-context/), [OTel — Context propagation](https://opentelemetry.io/docs/concepts/context-propagation/), [LangWatch — Trace IDs in AI](https://langwatch.ai/blog/trace-ids-llm-observability-and-distributed-tracing)

## R2-3. 대시보드 계층화 방법론 — Golden Signals / RED / USE (+ LLM 특수성)
- **Four Golden Signals**: latency·traffic·errors·**saturation**. **RED**(Rate·Error·Duration, 서비스 관점) + **USE**(Utilization·Saturation·Error, 리소스 관점). "Golden/RED = 사용자 경험이 좋은가", "USE = 서버가 건강한가" → 둘 다 쓴다.
- **LLM 서빙의 특수 해석(중요)**:
  - latency는 prompt 크기·출력 토큰 수에 따라 **본질적으로 가변** — *긴 응답이 곧 비정상은 아니다*. 단순 임계 알림이 오작동.
  - *지나치게 빠른 응답*은 **safety filter가 즉시 차단**한 신호일 수 있음 → 낮은 latency가 오히려 **높은 거부율**을 의미할 수 있다. (R3에서 semantic-router/guardrail 메트릭과 연결)
  - saturation: 전통적 CPU/mem/disk 대신 **GPU VRAM** — 추론엔진이 공격적으로 **사전할당**하므로 VRAM 사용률을 그대로 saturation으로 읽으면 오독.
- → **설계 함의**: 대시보드 L1을 **RED(서비스) + USE(GPU)** 두 축으로 구성하되, LLM 특수 카드(토큰정규화 latency, 거부율, KV/VRAM 사전할당 보정)를 둔다.
- 참고: [4 Golden Signals(Site24x7)](https://www.site24x7.com/learn/4-golden-signals.html), [Golden/RED/USE 정리(Medium)](https://medium.com/@farhanramzan799/mastering-observability-in-sre-golden-signals-red-use-metrics-005656c4fe7d), [비결정 인프라의 텔레메트리(DevOps.com)](https://devops.com/the-death-of-the-four-golden-signals-designing-telemetry-for-non-deterministic-infrastructure/)

## R2-4. 비용 귀속(FinOps) — groupby의 비즈니스 버전
- **showback(보여주기) → chargeback(실제 청구)**. 대부분 showback부터. 입자: **per-namespace / per-tenant / per-token / per-experiment**.
- **구현 패턴**: 게이트웨이(LiteLLM/Kong/Nginx)가 매 요청에 **team/project 헤더 주입** → vLLM 토큰 카운터를 Prometheus로 export → Grafana에서 GPU-hour 비용과 토큰을 join → **귀속비용 = (team_tokens / total_tokens) × gpu_hour_cost**.
- **함정**: 태그 전파가 인스턴스에서 멈춘다 — k8s pod의 `team=nlp` 라벨이 **DCGM GPU 메트릭엔 기본적으로 안 붙는다**. 멀티테넌트 vLLM pod 1개가 5개 팀을 서빙하면 청구서엔 한 줄로 보임. → **명시적 차원 주입 설계가 필요**.
- → **설계 함의**: 우리 공통 차원(model/endpoint/namespace)에 **tenant/feature**를 1급 차원으로 추가. 비용 대시보드 L2 = tenant×model groupby, 드라이버는 cache hit(R1-2)·토큰량.
- 참고: [Spheron — GPU FinOps 할당/차지백](https://www.spheron.network/blog/gpu-cloud-finops-ai-teams-cost-allocation-chargeback-budgeting/), [zop.dev — per-feature 비용 귀속](https://zop.dev/resources/blogs/llm-finops-per-feature-token-budget/), [Sysart — 온프레 토큰 예산/귀속](https://sysart.consulting/insights/token-budget-management-on-premises-llm/), [FinOps for AI(FinOps Foundation)](https://www.finops.org/wg/finops-for-ai-overview/)

## R2-5. ★ 시맨틱 레이어 — AI 해석을 신뢰성 있게 만드는 grounding
- 시맨틱 레이어 = 메트릭·차원·엔티티 관계의 **거버넌스된 정의**. LLM의 일은 자연어 질문을 **"올바른 메트릭+차원 조합으로 분해"** 하는 것으로 축소되고, 실제 쿼리 생성은 결정적 엔진(dbt MetricFlow 등)이 담당.
- **정확도 근거**: 시맨틱 레이어+비즈니스 컨텍스트가 있으면 NL 질의 정확도 **86–95%**, 컨텍스트 없는 raw DB는 **50–70%**.
- → **설계 함의(우리 MCP의 핵심)**: §2.3의 "인사이트 동사 tool"은 그 자체로 **시맨틱 레이어**다. 메트릭마다 (이름·의미·단위·groupby 가능 차원·관련 메트릭)을 정의한 **메트릭 카탈로그를 MCP resource**로 노출하면, AI가 자유 PromQL을 짜는 대신 검증된 메트릭/차원 조합을 고르게 되어 환각이 줄고 정확도가 오른다. "AI가 대시보드를 *신뢰성 있게* 해석"의 전제 조건.
- 참고: [dbt — Semantic Layer as Data Interface for LLMs](https://www.getdbt.com/blog/semantic-layer-as-the-data-interface-for-llms), [Cube — Semantic Layer & AI](https://cube.dev/blog/semantic-layer-and-ai-the-future-of-data-querying-with-natural-language), [AtScale — semantic context for NLQ](https://www.atscale.com/blog/semantic-context-natural-language-query/), [Databricks — Semantic Layer 아키텍처](https://www.databricks.com/blog/semantic-layer-architecture-components-design-patterns-and-ai-integration)

## R2-6. AIOps — 이상탐지·근본원인을 AI가 (자동 groupby/해석의 완성형)
- **이상탐지 기법**: STL 분해 + robust z-score, 계절 baseline은 Prophet/STL-LOESS, 상관은 Pearson/상호정보량(최근 N분), **근본원인 랭킹 = 가중 그래프 탐색 + 벤더 휴리스틱**. 규칙기반과 달리 "정상"을 학습해 이탈을 탐지.
- **LLM 역할**: 상위 후보를 읽고 **사람이 읽을 맥락을 작성**, postmortem 초안까지. **HolmesGPT**(CNCF sandbox AI SRE): Alertmanager/PagerDuty 알림 수신 → 클러스터의 로그·메트릭·이벤트 수집 → LLM이 근본원인 진단 + 런북 제안.
- 효과: 알림 노이즈 70–95%↓, Sev-2 MTTR 20–40%↓.
- → **설계 함의**: 우리 MCP의 `top_outliers`(BubbleUp형)와 결합하면 "이상 그룹 자동 탐지 → AI가 자연어 RCA"의 풀 파이프라인. HolmesGPT 툴셋 패턴을 R3에서 더 본다.
- 참고: [AI Anomaly Detection 가이드(OpenObserve)](https://openobserve.ai/blog/ai-anomaly-detection-guide/), [awesome-LLM-AIOps](https://github.com/Jun-jie-Huang/awesome-LLM-AIOps), [AIOps Kubernetes 2026](https://devops.gheware.com/blog/posts/aiops-kubernetes-observability-2026.html)

# 심화 리서치 Round 3 — 에이전트형 RCA · trace 역추적 · 동적 groupby UX · grounding의 한계

> 파생 아이디어: R2에서 "단일 trace id + 시맨틱 레이어 + AIOps"가 모이면 AI가 대시보드를 능동 해석할 수 있음을 봤다. R3는 이를 **구현 가능한 부품**으로 분해한다: ① 에이전트가 어떻게 tool을 골라 조사하나(HolmesGPT), ② exemplar trace id로 어떻게 trace를 역추적하나(Langfuse), ③ "클릭→groupby"를 어떻게 구현하나(Grafana), ④ AI에게 자유 쿼리를 주면 왜 위험한가(text-to-PromQL 한계).

## R3-1. ★ HolmesGPT — 에이전트형 조사 루프의 청사진
- **구조**: LLM + 라이브 관측 데이터를 잇는 **agentic loop**(`ToolCallingLLM`). **ReAct 패턴** — 알림을 읽고 → tool 선택 → 결과 읽고 → 다음에 뭘 볼지 결정. 경로가 스크립트가 아니라 *모델이 발견한 것*에 따라 달라짐.
- **Toolset 추상화**: 관련 tool 묶음 = 데이터소스/능력 단위. 예) Kubernetes(logs/events/manifest), Cloud(MCP: AWS/Azure/GCP), **Observability(Prometheus/Datadog/New Relic/Grafana)**, Utility(bash/검색).
- **조사 과정**: Prometheus 쿼리 실행 + K8s 이벤트/로그 수집 + pod spec 점검 + 배포 이력 확인을 **반복 호출·상관**하여 일관된 RCA 생성. "정적 대시보드/챗봇과 달리 능동적으로 무엇을 가져올지 결정".
- → **설계 함의**: 우리 **FABRIX MCP의 인사이트-동사 tool들이 곧 HolmesGPT의 Toolset**. 동일 패턴(ReAct + Toolset)으로 "TTFT 이상 → groupby_metric(model) → top_outliers → trace_drilldown → 자연어 RCA" 자동 조사 에이전트를 우리 도메인에 구성 가능. (오픈소스를 그대로 붙이거나, 패턴만 차용)
- 참고: [CNCF — HolmesGPT](https://www.cncf.io/blog/2026/01/07/holmesgpt-agentic-troubleshooting-built-for-the-cloud-native-era/), [DeepWiki — robusta-dev/holmesgpt](https://deepwiki.com/robusta-dev/holmesgpt), [GitHub HolmesGPT](https://github.com/HolmesGPT/holmesgpt), [holmesgpt.dev](https://holmesgpt.dev/dev/why-holmesgpt/)

## R3-2. Langfuse trace 역추적 — exemplar trace id → trace 상세 (L3 구현)
- **trace_id 규격**: 32 hex 문자. `start_as_current_observation`의 `trace_context`에 trace_id 지정 가능 → **외부에서 만든 trace id로 Langfuse trace를 시작/조회**할 수 있다(=게이트웨이 traceparent의 trace id를 그대로 사용).
- **Sessions**: `sessionId`(<200 ASCII)로 여러 trace를 묶어 **세션 리플레이**. 멀티턴/멀티스텝 에이전트에 적합.
- **속성 전파**: metadata 키(generation_name·trace_id·session_id·tags…)는 `langfuse` 접두 OTel span 속성으로 export → 관측 백엔드에서 검색/필터. 모든 span에 공통 차원(model/tenant)을 싣는 권장 방법 = **OTel Baggage + BaggageSpanProcessor**.
- → **설계 함의**: R1-5의 exemplar trace id == Langfuse trace_id 로 정합 → 대시보드 메트릭 점 → 정확한 Langfuse trace(span별 token/cost/latency/score)로 직행. 공통 차원은 Baggage로 전 span 자동 부착.
- 참고: [Langfuse — Sessions](https://langfuse.com/docs/observability/features/sessions), [Langfuse — OTel](https://langfuse.com/integrations/native/opentelemetry), [LiteLLM — Langfuse OTel](https://docs.litellm.ai/docs/observability/langfuse_otel_integration)

## R3-3. "클릭 → groupby"의 구현 — Grafana Filter & Group by (이미 제품화됨)
- **Grafana 13.1**: ad-hoc filter가 **"Filter and Group by"** 로 통합·확장. **데이터소스에서 가능한 차원을 자동 조회**해 사용자가 필터/그룹을 동적으로 추가. (차원마다 변수를 미리 만들 필요 없음)
- **drill-down UX**: 패널 **툴팁/시리즈 라벨 클릭 → 즉석 필터 in/out**, 패널 헤더의 **group-by 액션 → 그 자리에서 새 차원으로 분해**, 그리고 *그 그룹핑이 대시보드 전 패널에 반영*. 대시보드 레벨에서 활성 필터/그룹을 한눈에 관리.
- → **설계 함의**: 우리가 원한 "첫 화면 추상 → 클릭하면 groupby로 더 구체"의 **정확한 레퍼런스 구현**. 우리 React 대시보드도 (a) 차원을 백엔드가 자동 노출, (b) 차트 요소 클릭 = 필터/그룹 토글, (c) 선택이 전 패널 동기화 — 이 3원칙으로 만든다. (백엔드 §3.4 공통 차원 메타데이터가 이걸 가능케 함)
- 참고: [Grafana — Faster exploration: quick filters & grouping](https://grafana.com/whats-new/2026-04-14-faster-dashboard-exploration-with-quick-filters-and-grouping/), [Grafana — Add variables](https://grafana.com/docs/grafana/latest/visualizations/dashboards/variables/add-template-variables/), [Grafana Scenes — Adhoc filters](https://grafana.com/developers/scenes/advanced-adhoc-filters)

## R3-4. Semantic Router / Guardrail 메트릭 — "빠른 응답 = 거부"의 해소
- LLM Semantic Router는 **Prometheus 메트릭 + Grafana**로 라우팅 관측: latency·throughput·error, TTFT·ITL까지. 라우팅(콘텐츠 분석)은 **50–100ms** 추가 → 텔레메트리는 **비동기로 기록**, hot path에선 외부 호출 금지(관측이 병목 되면 안 됨).
- **end-to-end 커버리지** = 게이트웨이+모델+retrieval+tools+**guardrails** 데이터를 매 요청마다. 거부는 토큰 rate-limit의 rejected count 등으로 추적.
- → **설계 함의**: R2-3의 "빠른 latency가 거부일 수 있다"를 해소하려면 **거부/차단/라우팅 결과를 1급 차원**으로 둔다(우리 [semantic-router.md](../integration/semantic-router.md) 연동과 직결). L1 카드에 "정상응답 vs 차단/거부" 분리 표시.
- 참고: [TrueFoundry — AI Gateway 관측 지표](https://www.truefoundry.com/blog/observability-in-ai-gateway), [Red Hat — LLM Semantic Router](https://developers.redhat.com/articles/2025/05/20/llm-semantic-router-intelligent-request-routing), [Portkey — LLM 관측 가이드 2026](https://portkey.ai/blog/the-complete-guide-to-llm-observability/)

## R3-5. ★ Text-to-PromQL의 한계 → "메트릭 카탈로그 grounding"이 필수
- **정확도 현실**: PromAssistant/PromCopilot 등 LLM text-to-PromQL은 **GPT-4 백본으로도 ~69%**. 자연어의 모호함 vs PromQL의 정밀함이 충돌하고, **메트릭 정의가 맥락 의존적**(같은 "CPU"라도 서비스마다 의미가 다름)이라 환각이 잦다.
- **해결책 = grounding**: 메트릭 지식베이스 **RAG** 또는 **하이브리드 메트릭 카탈로그**(정적 큐레이션 ~2,000 메트릭 + 런타임 하드웨어 디스커버리; 3개 GPU 벤더 커버)로 정확한 메트릭 메타데이터를 추천.
- → **설계 함의(결정적)**: *AI에게 자유 PromQL/MetricsQL을 짜게 하지 말 것.* 대신 우리가 큐레이션한 **메트릭 카탈로그(이름·의미·단위·groupby 차원·관련 메트릭·맥락)를 MCP resource**로 주고, AI는 "검증된 메트릭+차원 조합 선택 + 인사이트-동사 tool 호출"만 하게 한다. 이게 R2-5(시맨틱 레이어, 86–95%)와 합쳐져 **신뢰성 있는 AI 해석**의 핵심. (= Datadog의 structured-output grounding, §2.2와 같은 철학)
- 참고: [PromAssistant: Text-to-PromQL (arXiv 2503.03114)](https://arxiv.org/html/2503.03114v2), [Catalog-Driven NL→PromQL (arXiv 2604.13048)](https://arxiv.org/html/2604.13048), [Alibaba — AI agents for PromQL](https://www.alibabacloud.com/blog/dont-understand-promql-ai-agents-help-you-with-large-scale-metric-data-analysis_602420)

## R3-6. 통합 트레이스+메트릭 단일창 — 제품 트렌드 확인 (MLflow 2026 등)
- 2026 LLM 관측 트렌드: **trace(span 디버깅) + metrics를 한 화면**에. 에이전트 span 분해(sub-agent/tool/retrieval/LLM completion)에 **token·latency·cost를 인라인** 표기(쿼리 없이 비용 귀속). PM/QA 등 **비개발자도 trace 주석·평가**에 참여(협업 워크플로).
- → **설계 함의**: 우리 L3(trace 상세)도 span 트리에 token/cost/latency/score 인라인. observe 프로파일(삼성증권)에서도 **비개발자가 읽을 수 있는** 자연어 요약·주석을 1급 기능으로.
- 참고: [MLflow — LLM Observability UI 2026](https://mlflow.org/articles/llm-observability-with-the-best-ui-a-2026-engineers-guide/), [Datadog — Agent Observability](https://docs.datadoghq.com/llm_observability/)

# 종합 — 3라운드가 수렴한 아키텍처 (§5 갱신)

3라운드 리서치가 하나의 그림으로 수렴한다. **"단일 trace id"** 를 척추로, **공통 차원**을 갈비뼈로, **메트릭 카탈로그(시맨틱 레이어)** 를 AI 해석의 grounding으로 삼는다.

```
[L1 Overview]  RED(서비스)+USE(GPU) 요약 · 정상/거부 분리 · 이상 Top-N         ← 추상
     │ 클릭(시리즈/카드)
[L2 Group]     공통차원 groupby(model×endpoint×namespace×tenant×time)         ← 구체
     │         자동 outlier 강조(BubbleUp형 top_outliers) · 병목 판정카드
     │ 클릭(데이터점 = exemplar)
[L3 Detail]    Langfuse trace 상세(span별 token/cost/latency/score) · 세션리플레이  ← 최구체
```
세 층을 관통하는 두 축:
- **차원 축(groupby)**: `model · endpoint · namespace · tenant · status · time` (+ Langfuse: prompt version, user/session는 필터 전용)
- **상관 축(drill-through)**: 게이트웨이 **traceparent → 단일 trace id** → 메트릭 **exemplar** → Langfuse **trace_id** (Baggage로 공통차원 전파)

**AI 해석(MCP) 레이어**: 메트릭 카탈로그(resource) + 인사이트-동사 tool + ReAct 조사 루프(HolmesGPT 패턴) + read-only(observe)/full(manage) 권한.

### 갱신된 적용 권장 (구현 부품 단위)
| # | 부품 | 근거 | 비고 |

## C. 핵심 의사결정 결론 (코드 근거)
- **D1 exemplar = 채택 안 함(확정).** VictoriaMetrics는 exemplar 미지원([VM 문서/이슈](https://github.com/VictoriaMetrics/VictoriaMetrics/issues/9592)). → **대안: 공통 차원 + 시간窓 조인.** "TTFT가 튄 (model, time)" → Langfuse `listObservations(model, time-range, decision...)`로 trace 목록. glue가 이미 model/time/decision/trace id를 적재하므로 **추가 적재 없이 조인 가능**. (exemplar보다 정밀도는 낮지만 구현 단순·VM 제약 회피)
- **D2/D3 trace id 일원화 = 거의 됨.** glue가 traceparent→32hex 정규화 완료. 남은 확인: **게이트웨이/어댑터가 traceparent를 capture 페이로드에 실어주는가**(어댑터 책임). 메트릭(VM)엔 trace id 부재가 정상 — 조인은 차원+시간으로.
- **D4 MCP 역할 분리 = 명확히 가능.** Langfuse 네이티브 MCP(trace/score/관측)는 **그대로 활용**. FABRIX MCP는 **Dynamo/VM 메트릭 + 대시보드 의미 + cross-source groupby + 메트릭 카탈로그**만 담당. 중복 없음.
- **D5 read-only = 기존 capability 재사용.** observe/manage capability 게이팅([capability.go](../../backend/internal/capability/capability.go))을 MCP tool 노출에도 적용 → Grafana `--disable-write` 동급.

## D. 권장 개발 순서 (레버리지 順)
1. **[중·백엔드] 공통 차원 groupby를 메트릭 경로에 도입.** live provider 쿼리에 `by (model[,endpoint])` 추가, domain 타입에 endpoint/namespace 차원. → 즉시 L2 groupby 가능. *가장 레버리지 큼, UI 변경 전 선행.* ✅ **구현 완료(2026-06-29)** — 아래 E 참조.
2. **[중] 메트릭 카탈로그 정의**(Go 단일 출처 + JSON 노출): 메트릭별 이름·단위·차원·의미·관련메트릭. drill UI와 MCP가 공유. R3-5 grounding 자산.
3. **[중·프론트] L1/L2/L3 + 동적 groupby UI.** 차트 요소 클릭=차원 토글, 선택 전 패널 동기화(Grafana 13.1 패턴, R3-3). GPU 3단을 다른 화면으로 일반화.
4. **[중] metric→trace 조인 drill-through.** (model,time[,decision]) → Langfuse observations. L2→L3 완성.
5. **[중~고] FABRIX MCP 서버.** 카탈로그 resource + 인사이트-동사 tool(`summarize_endpoint_health·groupby_metric·top_outliers·explain_panel·trace_drilldown`), capability 게이팅. Langfuse MCP와 병렬.
6. **[선택] top_outliers/ReAct RCA**(HolmesGPT 패턴), Langfuse OTLP 전환.

> 검토 결론: 리서치 목표의 **하부 구조(멀티소스·Dynamo 메트릭·trace id·capability·일부 drill)는 이미 있다.** 빠진 건 ① 메트릭의 차원 groupby, ② 통일된 계층/동적 UI, ③ AI grounding(카탈로그)+FABRIX MCP. **1→2→3 순으로 가면 UI·AI 양쪽이 같은 차원/카탈로그 자산을 재사용**하므로 중복이 없다.
