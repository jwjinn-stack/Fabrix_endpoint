# 챗봇/사내서비스 End-to-End 레이턴시 측정 — 관측 소프트웨어 리서치

> **질문(고객 요구)**: 챗봇·사내 서비스에서 **애플리케이션에서 출발한 레이턴시**와, 챗봇 응답을 주기까지 **그 사이사이(각 구간/hop)의 레이턴시**를 모두 측정·표시하고 싶다. 이를 가능케 하는 사용 소프트웨어가 무엇인지 조사.
> **작성일**: 2026-06-20 (Asia/Seoul) · 회사: 메이머스트(MAYMUST)
> **짝 문서**: [기능-현황-로드맵.md](기능-현황-로드맵.md) · [인프라-통신-아키텍처.md](인프라-통신-아키텍처.md) · [경쟁솔루션-벤치마킹.md](경쟁솔루션-벤치마킹.md)
> **출처 표기 규칙**: 〔공식문서〕 = 1차 공식 문서 확인 · 〔리서치〕 = 비교/벤더 글 기반(확인 필요) · 〔현코드〕 = 본 레포 구현 확인.

---

## 0. 한 줄 결론

고객이 말하는 **"애플리케이션부터의 레이턴시 + 사이사이 레이턴시"**는 관측(observability) 용어로 정확히 **분산 트레이싱(Distributed Tracing)** = 하나의 요청이 만드는 **스팬(span) 워터폴**입니다.
그리고 이건 우리가 로드맵에서 여러 번 `MAY(victoria-traces 미수집)`로 남겨둔 **바로 그 갭**이며, **현재 스택(NVIDIA Dynamo + vLLM)이 이미 OTLP·W3C 트레이스를 네이티브 지원**합니다〔공식문서〕. 즉 신규 벤더 도입이 아니라 **이미 가진 것을 켜고 수집·시각화 백엔드만 붙이면** 되는 일입니다.

---

## 1. 요구를 아키텍처에 매핑 — "사이사이"가 정확히 어디인가

우리 [인프라-통신-아키텍처.md](인프라-통신-아키텍처.md) §5(B) 추론 흐름에 고객 요구를 겹치면, 측정해야 할 **구간(hop)**은 다음과 같습니다:

```
[챗봇/사내앱] ─① 네트워크 ─▶ [web/nginx] ─② ─▶ [BFF(Go :8080)]
   ─③ classify ─▶ [가드레일 Semantic Router]
   ─④ (운영경로) ─▶ [Envoy Gateway]
   ─⑤ ─▶ [Dynamo Frontend] ─⑥ ─▶ [KV Router] ─⑦ ─▶ [Prefill Worker]
   ─⑧ ─▶ [Decode Worker(vLLM)] ─⑨ 응답 스트림 ─▶ ... ─▶ [챗봇]
```

고객이 원하는 건 위 ①~⑨ **각 화살표마다 걸린 시간**을 한 요청 단위로 쪼개 보는 것입니다. 이를 제공하는 SW는 측정 **레이어별**로 6종류로 나뉩니다(§2~§7).

| 측정 대상 구간 | 담당 소프트웨어 계층 | 본 문서 절 |
|---|---|---|
| 표준·프로토콜(공통 언어) | OpenTelemetry + GenAI Semantic Conventions | §2 |
| 트레이스 저장·워터폴 시각화 | VictoriaTraces / Grafana Tempo / Jaeger | §3 |
| ⑤~⑨ 추론 엔진 내부(큐·prefill·decode) | vLLM, NVIDIA Dynamo 네이티브 트레이싱 | §4 |
| ④ 게이트웨이/메시 구간 | Envoy / Istio 트레이싱 | §5 |
| ①~② 앱·서비스 구간(무코드) | eBPF (Beyla/OBI, Pixie, Cilium Hubble) | §6 |
| 앱 시맨틱(프롬프트·토큰·평가) | Langfuse / Phoenix / Helicone / Datadog 등 | §7 |

## 2. 표준·프로토콜 계층 — OpenTelemetry (모든 것의 공통어)

분산 트레이싱의 사실상 표준. **OTLP**(OpenTelemetry Protocol, HTTP/gRPC)로 스팬을 보내고, **W3C Trace Context**(`traceparent` 헤더)로 요청이 컴포넌트를 넘어가도 **하나의 trace_id로 이어 붙입니다**〔공식문서〕. 컨텍스트 전파가 안 되면 트레이스가 끊겨 "사이사이"가 안 보입니다 — 이게 가장 중요한 전제.

- **GenAI Semantic Conventions**: LLM 워크로드 전용 표준 스키마(프롬프트·모델·토큰 사용량·도구/에이전트 호출). 이 덕에 LangChain 에이전트 스팬과 원본 OpenAI 호출 스팬이 동일하게 보임. Google Cloud·AWS·Azure·Datadog 등이 채택〔리서치〕.
- ⚠ **상태 주의**: GenAI/MCP semantic conventions는 2026-05 기준 여전히 **Development(개발) 상태**, 안정화 타임라인 미공개 → 속성명이 바뀔 수 있음(확인 필요)〔리서치〕.

> 시사점: 우리가 무엇을 채택하든 **OTLP + W3C 전파**를 중심에 두면 벤더 종속 없이 ①~⑨를 한 trace로 묶을 수 있음.

## 3. 트레이스 백엔드 — 스팬을 저장하고 "워터폴"로 보여주는 곳

수집된 스팬을 저장하고, 요청 1건의 구간별 막대(워터폴/플레임그래프)로 그려주는 DB. 우리 화면(`Traffic.tsx`의 Tree↔Waterfall 토글)이 노리던 per-request 뷰가 여기서 나옵니다.

| 백엔드 | 강점 | 우리 적합도 |
|---|---|---|
| **VictoriaTraces** | VictoriaMetrics 팀 제작. Tempo 대비 **RAM 3.7배·CPU 2.6배 절감**, 외부 스토리지(객체스토리지/ClickHouse/ES) 불필요, OTLP 수집 + Jaeger 쿼리 호환〔리서치〕 | ★ **최적** — 우리가 이미 VictoriaMetrics 스택. 로드맵에 적은 `victoria-traces`가 바로 이것 |
| **Grafana Tempo** | 인덱싱 생략(Trace ID 조회), 대규모 단순. OTLP/Jaeger/Zipkin 수집. **NVIDIA Dynamo 공식 트레이싱 가이드가 Tempo 기준**〔공식문서〕 | ★ 강력 — Dynamo 문서 그대로 따라가면 즉시 동작, Grafana 시각화 |
| **Jaeger** | CNCF Graduated, 성숙. Cassandra/ES 백엔드 필요〔리서치〕 | △ 가능하나 별도 스토리지 운영부담 |

> 권고: **VictoriaTraces(우리 스택 일관성) 또는 Tempo(Dynamo 공식 경로)** 중 택1. 둘 다 OTLP 수집·Grafana 시각화.

## 4. 추론 엔진 내부(⑤~⑨) — 이미 우리 스택이 지원 ★핵심

고객 요구의 "마지막 사이사이"(큐 대기 → prefill(TTFT) → decode)는 **추가 벤더 없이** 현 엔진이 냅니다.

### 4-1. vLLM 〔공식문서〕
- **메트릭(Prometheus)**: `vllm:time_to_first_token_seconds`(TTFT), `vllm:inter_token_latency_seconds`(TPOT/ITL), `vllm:e2e_request_latency_seconds`(종단간), `vllm:request_queue_time_seconds`(큐 대기), `vllm:request_prefill_time_seconds`, `vllm:request_decode_time_seconds`, `vllm:num_requests_running/waiting`, `vllm:kv_cache_usage_perc`.
  - → 우리는 이미 이 다수를 vmselect로 읽어 `LatencyBreakdown`·`SchedulerState`로 쓰는 중(v0.16.0)〔현코드〕. **집계(평균/백분위)** 수준.
- **분산 트레이싱**: `--otlp-traces-endpoint`로 OTLP export 지원. 스케줄러 큐 구간·inference·prefill·decode **스팬 타이밍** 기록. `--collect-detailed-traces`로 model forward/execute까지(성능 비용 주의)〔공식문서〕.
  - → 이걸 켜면 집계가 아닌 **요청 1건의 구간 스팬**이 나옴 = 고객이 원하는 per-request 분해.

### 4-2. NVIDIA Dynamo 〔공식문서〕
- **W3C Trace Context를 파이프라인 전체에 전파** → frontend·router·worker·엔진 스팬을 **하나의 trace로 상관(correlate)**.
- **캡처되는 스팬(예시)**: Frontend `http-request`(루트) → KV Router `route_request`/`select_worker`/`find_matches`/`schedule` → Worker `handle_payload` → 엔진 `tokenize`/`prefill_forward`/`decode_forward`/`decode_loop`.
- **수집**: OTLP(gRPC :4317) → **Tempo/Grafana**. 활성화는 env(`OTEL_EXPORT_ENABLED=true`, `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`) + 워커 `--enable-trace`. 상세도는 `SGLANG_TRACE_LEVEL`(1~3, 레벨3은 ~160만 스팬/시간).
- **메트릭**: `dynamo_component_*`(요청수·처리시간·바이트), `dynamo_preprocessor_*`, 엔진 메트릭(vLLM/SGLang/TRT-LLM).

> **즉, ④Gateway~⑨Decode 구간의 per-request 워터폴은 "Dynamo 트레이싱 ON + OTLP→Tempo/VictoriaTraces" 만으로 표준 제공됨.** 이것이 로드맵 P4-3의 `MAY(분산 스팬 트리)`를 해제하는 정공법.

## 5. 게이트웨이/메시 구간(④) — Envoy / Istio

요청이 게이트웨이를 지날 때의 지연. 우리는 가드레일 운영 경로에 **Envoy Gateway + EnvoyExtensionPolicy**를 이미 씀(로드맵 v0.11.0).
- Envoy/Istio는 **B3·W3C Trace Context 헤더**를 확인/생성하고 사이드카가 자동으로 스팬 전송〔리서치〕. `x-request-id` 병행 권장.
- 컨텍스트 전파만 켜면 게이트웨이 hop이 같은 trace에 막대로 붙음 → 가드레일 ext_proc(Semantic Router) 오버헤드까지 한 화면에서 분해 가능.

---

## 6. 애플리케이션·서비스 구간(①~②) — eBPF 무(無)코드 계측

"챗봇/사내앱에서 출발한 레이턴시"를 **앱 소스 수정 없이** 커널 레벨에서 측정. 우리 BFF는 Go라 직접 OTel SDK도 쉽지만, 사내 챗봇/레거시 앱은 코드 접근이 어려울 수 있어 eBPF가 유효.

| 도구 | 성격 | 비고〔리서치〕 |
|---|---|---|
| **Grafana Beyla** → **OTel eBPF Instrumentation(OBI)** | 무코드로 HTTP/gRPC RED 메트릭 + OTel 스팬 발행 | 2026-04 KubeCon EU에서 Beyla 후계 **OBI 베타** 발표. CNCF |
| **Pixie** | 자동계측 APM 트레이스 + 서비스맵 | CNCF |
| **Cilium Hubble** | L3–L7 네트워크 가시성(서비스 간 통신/지연) | CNCF, Cilium CNI 위 |
| **Tetragon** | 보안 관측(런타임) | 보조 |

> 적합도: 사내 챗봇/앱이 다양·레거시면 **Beyla/OBI로 ①② 자동 스팬** → 같은 OTLP 파이프라인으로 합류. 코드 접근 가능한 신규 앱이면 OTel SDK 직접 계측이 더 정밀.

## 7. 앱 시맨틱 LLM 관측 플랫폼(프롬프트·토큰·품질) — 보완재

§4의 구간 지연과 달리, **"무엇을 물었고 토큰을 얼마 썼고 응답 품질은 어땠나"**를 트레이스에 얹는 층. 우리 평가(LLM-judge)·사용량 화면과 겹치되, 외부 베스트프랙티스 참고용.

| 도구 | 성격〔리서치〕 | 우리 관점 |
|---|---|---|
| **OpenLLMetry (Traceloop)** | OTel 위 LLM 자동계측(OpenAI/Anthropic/LangChain 등). self-host export 가능 | ★ OTel 네이티브라 우리 파이프라인에 합류 쉬움 |
| **Arize Phoenix** | OpenTelemetry 기반 오픈소스, 로컬/컨테이너 실행 | 자체호스팅 친화 |
| **Langfuse** | 최다 사용 OSS, 트레이싱+평가+프롬프트관리, self-host 가능. OpenLLMetry 연동 | 온프렘 적합, 평가 화면 레퍼런스 |
| **Helicone** | 프록시형(코드 2줄), MIT 오픈소스 | 우리 BFF가 이미 프록시 = 개념 일치 |
| **Datadog / Dynatrace LLM Observability** | APM에 LLM 스팬 통합, OTel GenAI semconv 네이티브 지원 | 상용·에이전트 필요 → 온프렘/에어갭엔 부적합 |

> 주의: Datadog 등 SaaS는 **온프렘/에어갭(증권사) 제약**과 충돌 가능. 우리 강점축(온프렘 WORM·4축 귀속)과 맞추면 **OSS·self-host(OpenLLMetry/Langfuse/Phoenix)** 우선.

## 8. 권고안 (현 스택 기준, 신규 벤더 최소화)

```
[사내앱/챗봇]
  └(eBPF: Beyla/OBI · 무코드 ①②)─┐
[web]→[BFF Go]                    │ OTel SDK 직접계측(③ 가드레일 포함)
  └ classify → [Semantic Router]  │
[Envoy Gateway](④ W3C 전파)       │  ─── 모두 OTLP ───▶  [VictoriaTraces or Tempo]
[Dynamo Frontend→Router→Worker]   │                         └▶ Grafana/우리 Traffic.tsx
  └ vLLM(큐·prefill·decode 스팬 ⑤~⑨)┘                            워터폴 per-request
```

**단계적 실행(낮은 리스크 → 높은 가치 순)**
1. **Dynamo/vLLM 트레이싱 ON** + **VictoriaTraces(또는 Tempo) 배포** → ④~⑨ per-request 워터폴 확보. *(로드맵 P4-3 `MAY` 해제의 정공법, 신규 벤더 0)*
2. **BFF(Go)에 OTel SDK** 추가 + **Envoy W3C 전파** → ③④ 구간(가드레일 오버헤드 포함)을 같은 trace에 결합. *(우리 `proxystats` 집계를 per-request 스팬으로 승격)*
3. 사내 챗봇/앱이 다양·레거시면 **Beyla/OBI(eBPF)** 로 ①② 무코드 합류.
4. (선택) 프롬프트·토큰·품질 시맨틱은 **OpenLLMetry/Langfuse**(self-host)로 보강 — SaaS(Datadog)는 온프렘 제약상 비권장.

**우리 화면 영향**: `Traffic.tsx`의 Tree↔Waterfall 토글이 드디어 **평균 집계가 아닌 실제 요청 1건의 ①~⑨ 막대**를 그릴 수 있게 됨. 대시보드 KPI(TTFT p95 등)는 메트릭(VictoriaMetrics) 유지, **드릴다운 시 trace_id로 워터폴 점프**가 이상적 UX.

## 9. 확인 필요 / 열린 질문
- 〔확인필요〕 사내 챗봇/앱의 런타임(Go/Python/Java/Node)과 **소스 접근 가능 여부** → SDK 직접계측 vs eBPF 선택을 가름.
- 〔확인필요〕 운영 클러스터에 **Tempo/VictoriaTraces 신규 배포** 가능 여부(스토리지·리소스) — 운영 승인 항목.
- 〔확인필요〕 Dynamo 트레이싱 상세도(`SGLANG_TRACE_LEVEL`) — 레벨3은 시간당 ~160만 스팬, 샘플링 정책 필요.
- 〔확인필요〕 GenAI semantic conventions가 아직 Development 상태 → 속성명 변경 리스크, 안정화 전엔 핵심 구간(지연)만 의존 권장.
