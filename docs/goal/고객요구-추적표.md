# FABRIX Endpoint — 고객요구 추적표 (RTM, Living)

> **목적**: 고객요구(R1~R11)를 **요구 중심**으로 추적한다. "각 요구가 어떤 기능으로, 어디까지 충족됐나".
> 기능 백로그([기능-백로그.md](기능-백로그.md))의 `🔴 고객요구` 플래그와 **R코드로 양방향 연결**:
> - 기능 백로그 = *기능 중심* (이 기능은 어떤 요구를 위한가)
> - 이 추적표 = *요구 중심* (이 요구는 어떤 기능들로 충족되나 + 상태)
> 원천: [고객요구사항정리.md](../고객요구사항정리.md)(러프) → [고객요구사항-vllm-개발매핑.md](../고객요구사항-vllm-개발매핑.md)(정규화 R1~R11).
> 범례: ✅ 충족 · 🟡 부분 · ⏳ 미착수 · 최종 갱신 2026-06-18

## 정리 규칙 (어떻게 관리하나)
1. **요구는 R코드로 고정**(R1~R11). 새 요구는 R12…로 추가하고 [고객요구사항-vllm-개발매핑.md] §1에도 반영.
2. 각 요구에 **충족 기능(들)** 을 백로그에서 연결 + **충족상태** 판정 + **근거**(무엇이 됐고 뭐가 남았나).
3. 기능을 만들 때 그 기능의 `🔴 고객요구` 플래그 ↔ 여기 표가 일치해야 함(불일치 시 양쪽 수정).
4. 상태는 기능 완료에 따라 갱신. **충족률**을 맨 아래에 집계.

## 1. 요구사항 추적 매트릭스

| R | 요구 (정규화) | 충족 기능 (백로그) | 상태 | 근거 / 남은 것 |
|---|---------------|--------------------|------|----------------|
| **R1** | 사용 주체 식별·귀속 (sessionId/DB ID/SI, **사내 DB 연동**) | identity-broker · RBAC/Users · 사용량 리포트 | 🟡 부분 | 설계상 user_ref(sessionID 해시)·employee_id 수용. **사내 DB 연동 미착수**(블로커: 연동 방식 확정). |
| **R2** | 사용현황·품질 (API별, **TTFT**, "얼마나 잘") | 관제 대시보드 · 사용량 리포트 · live 연동 | ✅ 충족 | live TTFT p50/p95·ITL·QPS 실측 표시. |
| **R3** | 출처(프론트/앱) 귀속 | 사용량 리포트 · rollup-worker · 키·앱 | 🟡 부분 | app_id=API키 매핑 구조 ✅. **앱축 집계는 rollup-worker 후**(현재 모델축만). |
| **R4** | API Key 단위 분석 | 엔드포인트 발급/키·앱 · 키 쿼터+스펜드 | 🟡 부분 | 키 발급·회수·해시저장 ✅. **키별 사용량 집계·쿼터 강제 ⏳**. |
| **R5** | 프록시/Pod 통신 가시화 | 트래픽/프록시 뷰 (4-5) | ⏳ 미착수 | victoria-traces 인프라 존재. span 타임라인 화면 미구현. |
| **R6** | 애플리케이션 레벨 모니터링 | 관제 대시보드(앱 분포) · rollup-worker | 🟡 부분 | 앱별 요청 분포 카드 ✅. 앱 단위 정밀 집계 rollup 후. |
| **R7** | GPU 효율·MIG 검증 ("얼마나 잘 짤랐나") | GPU/MIG 화면(4-4) · 대시보드 GPU 카드 | 🟡 부분 | DCGM GPU 메트릭 live 카드 ✅. **MIG 슬라이스 효율 화면·스코어 ⏳**(현재 MIG 미분할). |
| **R8** | 멀티 워크로드/멀티 모델 (챗봇·batch·agentic, gemma·qwen) | 모델 카탈로그 · 플레이그라운드 · 엔드포인트 생성 위저드 | ✅ 충족 | 카탈로그 5모델·플레이그라운드 실추론 ✅. 배포 위저드(CR)는 확장. |
| **R9** | 가드레일(AI) + **증적** (SDS/fluent) | 가드레일 배선 · audit-ingestor · 증적 뷰(4-3) · 한국어PII · 정책카탈로그 | 🟡 부분 | Semantic Router 설치 Ready ✅. **게이트웨이 배선·증적 적재·증적 화면 ⏳**(P1 최상단). |
| **R10** | 커스텀 관제 ("입맛대로") | 입맛대로 관제뷰 빌더 | ⏳ 미착수 | P3. 핵심 화면 안정화 후. |
| **R11** | 네트워크 요건 (온프렘/폐쇄망) | (전제) 워크로드 인클러스터 배포 · Harbor 미러 | ✅ 충족(전제) | 온프렘 K8s 위 동작 검증. 폐쇄망은 운영 전환 시 Harbor 전량 미러. |

> ⚠ 내 Notion `🔴 고객요구` 옵션은 현재 **R1~R10**. **R11 추가** 필요(원하시면 옵션 추가 + 해당 기능에 부착).

## 2. 충족률 집계 (2026-06-18)
- ✅ 충족: **R2 · R8 · R11** (3)
- 🟡 부분: **R1 · R3 · R4 · R6 · R7 · R9** (6) — 대부분 P1/P2 진행으로 완결
- ⏳ 미착수: **R5 · R10** (2)
- **핵심 미충족 = R9(가드레일 증적)** → P1 최상단 "가드레일 게이트웨이 배선"이 R9 완결의 시작점.

## 3. 미해결/확인 필요 (고객 확인 사항)
[고객요구사항-vllm-개발매핑.md §7] 과 동일:
- R1: 유저 1차 키(sessionId/DB ID/SI) + **사내 DB 연동 방식** 고객 확인.
- R7: MIG 프로파일·슬라이스 배치 기준 확인(DCGM 전제).
- R9: "SDS"의 정체(사내 보안 시스템?)·송출 포맷, **증적 보존기간·WORM·접근통제** 컴플라이언스 확인.
