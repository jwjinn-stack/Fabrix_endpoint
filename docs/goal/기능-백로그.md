## 2. 기능별 상세 (행 클릭 = 이 섹션)

> 형식: **필요성 / 참고(경쟁사·문서) / 우선순위 근거 / 고객요구 매핑**

### 가드레일 게이트웨이 배선 `P1` `🔴R9`
- **필요성**: 현재 추론은 client→Dynamo 직결이라 가드레일(PII/Jailbreak)을 안 거침. 증권사 컴플라이언스의 핵심 = 모든 요청을 가드레일 통과시키고 판정 헤더(`x-vsr-matched-*`)를 받는 것.
- **참고**: vLLM Semantic Router(설치 Ready), Kong/Portkey의 게이트웨이 가드레일 라우팅.
- **우선순위 근거**: SR이 이미 Ready. 증적(audit-ingestor)·증적뷰(4-3)의 **선행 조건**이라 P1 최상단.
- **고객요구**: R9(가드레일 증적), SDS(가드레일→fluent).

### audit-ingestor (증적 파이프라인) `P1` `🔴R9`
- **필요성**: 가드레일 판정을 불변 증적으로 남겨야 함(증권사 감사). 헤더→증적 스키마(2-2) 정규화→ClickHouse `guard_audit`(+후속 WORM).
- **참고**: SSOT Part 2, fluent(SDS 송출), Lakera 증적.
- **우선순위 근거**: 게이트웨이 배선 직후 데이터가 흐르므로 바로 이어야 함.
- **고객요구**: R9, "SDS: 가드레일 문제→fluent로 매트릭"(고객요구사항정리.md).

### 가드레일 증적 뷰 (4-3) `P1` `🔴R9`
- **필요성**: 차단/PII/JB 이력을 부서·앱별로 조회·필터·상세(trace_id·정책버전·마스킹샘플). 증권사 핵심 화면.
- **참고**: Nutanix·Lakera 증적 UI, SSOT 4-3.
- **우선순위 근거**: 증적 데이터(audit-ingestor) 적재 후 즉시 가치.
- **고객요구**: R9 "가드레일 결과 증적".

### rollup-worker (귀속 집계) `P1` `🔴R2·R3`
- **필요성**: 사용량 리포트의 **부서·앱·키 축**을 채우려면 트레이스/메트릭을 `(user×app×key×model×시간)`로 배치 집계해야 함(메트릭 라벨은 카디널리티 때문에 불가).
- **참고**: SSOT 3-5 usage_rollup, Langfuse/Helicone 사용량 귀속.
- **우선순위 근거**: 사용량 리포트(완료)의 dept/app 빈 축을 실데이터로.
- **고객요구**: R2(사용량), R3(프론트 귀속), R6.

### 키 쿼터(rpm/tpd) + 스펜드 `P1` `🔴R4` `※경쟁갭`
- **필요성**: 발급·회수(완료)에 더해 **키별 호출/토큰 한도 강제 + 사용량 집계**. 경쟁사(LiteLLM·Portkey·NAI) 표준인데 우리는 미적용.
- **참고**: LiteLLM 예산·rate limit, Portkey 키별 정책.
- **우선순위 근거**: 키 발급 완료 직후 확장. 경쟁 갭 1순위.
- **고객요구**: R4 "API키별".

### 엔드포인트 생성 위저드 (Dynamo CR) `P2` `🔴R8`
- **필요성**: 콘솔에서 모델 배포 = `DynamoGraphDeployment` CR 생성(패턴·replica·MIG·오토스케일). "배포→검증" 루프 완성.
- **참고**: Nutanix Create Endpoint, Backend.AI 5단 위저드, SSOT D-5.
- **우선순위 근거**: 카탈로그/플레이그라운드 다음의 자연스러운 확장. CR 조작은 권한·검증 필요라 P2.
- **고객요구**: R8(모델).

### GPU/MIG 화면 (4-4) `P2` `🔴R7`
- **필요성**: GPU별 MIG 슬라이스 점유·**효율 스코어(3-4)** — "얼마나 잘 잘랐나". 대시보드 카드 외 전용 화면.
- **참고**: DCGM(수집 중), Nutanix Infrastructure Summary, SSOT 4-4.
- **우선순위 근거**: DCGM 데이터 이미 VM에 있음 → 화면만.
- **고객요구**: R7 "gpu 사용 기준→MIG→얼마나 잘 짤랐는지".

### 트래픽/프록시 뷰 (4-5) `P2` `🔴R5` `※경쟁갭`
- **필요성**: 요청 1건의 span 타임라인(gateway→router→engine), 프록시 오버헤드. pod간 통신 흐름.
- **참고**: victoria-traces(OTLP), Langfuse/Helicone 트레이스, SSOT 4-5.
- **우선순위 근거**: 트레이스 인프라 존재. 경쟁사 표준 관측.
- **고객요구**: R5 "pod간 통신 흐름·프록시 성능 매트릭".

### 한국어 PII 룰 보강 `P2` `🔴R9`
- **필요성**: 기본 탐지율(PII 26.7%) 부족. 주민/계좌/여권 룰 + 임계치 PoC. 증권사 필수.
- **참고**: SSOT G7, Lakera.
- **우선순위 근거**: 가드레일 경로(P1) 가동 후 정확도 보강.
- **고객요구**: R9.

### 멀티모델 비교 + View code `P2` `※경쟁갭`
- **필요성**: 플레이그라운드에서 여러 모델 동시 비교 + curl/python/js 스니펫 자동생성("UI 실험→코드 복사→API").
- **참고**: Fireworks·Together·Langfuse·Databricks.
- **우선순위 근거**: 플레이그라운드(완료) 확장. 경쟁 갭.
- **고객요구**: –.

### 가드레일 정책 카탈로그 + 토글 `P2` `🔴R9` `※경쟁갭`
- **필요성**: PII 1축 외 다축 정책(jailbreak·intent·secrets…)을 카탈로그화하고 라우트/앱에 토글로 부착.
- **참고**: Portkey·Kong·NeMo Guardrails 정책 카탈로그.
- **우선순위 근거**: 가드레일 경로 가동 후 정책 관리 UX.
- **고객요구**: R9.

### RBAC/Users · 부서 매핑 `P2` `🔴R1`
- **필요성**: 사용자/역할 관리(AI/ML Admin·User·Super Admin), 부서·직원 매핑.
- **참고**: Nutanix Admin Configuration, Backend.AI Credentials.
- **우선순위 근거**: 멀티테넌시·귀속의 기반. identity-broker와 연계.
- **고객요구**: R1 "누가를 특정 — sessionId/DB ID/SI/사내DB 연동".

### identity-broker (사내 DB 귀속) `P2` `🔴R1`
- **필요성**: sessionID→직원/부서 소급 enrich(사내 인사 DB/SI 연동). 4축 귀속의 마지막 축.
- **참고**: SSOT G1, D-1.
- **우선순위 근거**: 사내 DB 연동 방식 확정 필요(블로커). RBAC와 함께.
- **고객요구**: R1.

### 입맛대로 관제뷰 빌더 `P3` `🔴R10`
- **필요성**: 사용자가 패널·필터를 조합해 커스텀 관제뷰 구성.
- **참고**: Grafana 빌더, SSOT R10.
- **우선순위 근거**: 핵심 화면 안정화 후.
- **고객요구**: R10 "어플리케이션 통해 관제 입맛대로".

### 프롬프트/평가 관리 `P3` `※경쟁갭`
- **필요성**: 프롬프트 버전 관리 + LLM-as-judge·데이터셋으로 응답품질 회귀. 폐쇄망에서도 수요.
- **참고**: Langfuse·Databricks.
- **우선순위 근거**: 관측/거버넌스 성숙 단계.
- **고객요구**: –.

> 그 외(모델 임포트·알림 드로어·ObjectScale WORM·disagg/NIXL·다크모드)는 §1 표 참고. 착수 시 이 섹션에 상세 추가.
