## 2. 구현 노트 (리서치 기반 — 왜/어떻게/무슨 기술)

> 형식: **왜** / **경쟁사는 이렇게** / **우리 구현 접근(검증된 사실)** / **검증** / **iteration 포인트**

### [1] 가드레일 게이트웨이 배선 `P1` `🔴R9`
- **왜**: 현재 추론은 client→Dynamo 직결 → 가드레일 미경유. 증권사 컴플라이언스 = 모든 요청이 PII/Jailbreak 판정을 거치고 그 결과가 헤더로 남아야 함. R9(가드레일 증적) 전체의 입구.
- **경쟁사는 이렇게**: Kong/Portkey는 게이트웨이에 가드레일 플러그인을 라우트 단위로 부착. NeMo Guardrails는 정책 레일을 요청 경로에 삽입. → 공통점: **게이트웨이가 차단 지점**.
- **우리 구현 접근 (실측 사실)**:
  - SR 이미 Ready: ns `vllm-semantic-router-system`, svc `semantic-router:50051`(gRPC ext_proc)·`:8080`(HTTP classify).
  - 공유 Envoy Gateway(v1.7.2, `envoy` 클래스)는 **KServe만** 사용 → 우리 전용 **새 Gateway**를 `envoy` 클래스로 만들고 거기에만 ext_proc 부착(다른 팀 무영향, 리서치로 확인).
  - **EnvoyExtensionPolicy(per-route)** 채택 — 전역 `enableEnvoyPatchPolicy` 불필요(공유 컨트롤러 미변경). targetRef=우리 HTTPRoute, extProc backendRef=`semantic-router:50051`.
  - 구성: 우리 Gateway + HTTPRoute(→ `gemma4-31b-vllm-agg-frontend-nodeport.dynamo-inference:8000`) + EnvoyExtensionPolicy + 교차-ns ReferenceGrant(dynamo-inference, vllm-semantic-router-system).
  - 실제 헤더(소스 확인): `x-vsr-matched-pii`·`x-vsr-matched-jailbreak`·`x-vsr-selected-{category,model,confidence}`. ⚠ SSOT의 `x-vsr-pii-violation`은 부재 → 매핑 시 실제 이름 사용.
- **검증**: PII 포함 요청(예 "주민번호 123456-1234567") → 응답 헤더에 `x-vsr-matched-pii` + 차단. 클린 요청 → 통과.
- **iteration 포인트**: gRPC h2c 설정(BackendTrafficPolicy 필요할 수 있음), 차단 시 사용자 반환 메시지(content_filter_refusal), 라우팅을 gemma4 외 vllm 모델로 확장.

### [2] audit-ingestor (증적 파이프라인) `P1` `🔴R9` `←1`
- **왜**: 가드레일 판정을 **불변 증적**으로. 누가·언제·무엇이 차단됐나 = 증권사 감사 핵심.
- **경쟁사는 이렇게**: Lakera/Portkey는 모든 가드레일 이벤트를 로그/대시보드로 적재. → 우리는 여기에 **WORM 보존**까지(능가축).
- **우리 구현 접근 (SSOT 2-6 의사코드 그대로)**:
  - 위치: Envoy `ext_proc` access-log 또는 응답 후크에서 이벤트 수신(핫패스 비차단, async 큐+배치 flush).
  - 정규화: `x-vsr-matched-*`/`x-vsr-selected-*` → 증적 스키마(2-2). `user_ref`=salted SHA-256(x-user-id), 원문 키·PII 미저장(마스킹 샘플만).
  - 적재: **ClickHouse `guard_audit`**(이미 생성됨: ns fabrix-endpoint, svc `clickhouse:9000`, db fabrix, 스키마 2-4) + (후속) Fluent Bit→ObjectScale WORM.
  - 구현체: Go 워크로드(우리 레포 신규 `cmd/audit-ingestor` 또는 api에 후크). pgx 패턴 재사용, ClickHouse는 HTTP(:8123) 또는 native(:9000).
- **검증**: 가드레일 차단 발생 → `SELECT * FROM fabrix.guard_audit` 에 행 적재(해시·마스킹 확인, 원문 없음).
- **iteration 포인트**: 통과(allowed) 샘플링률, 배치 크기/주기, trace_id 조인(victoria-traces), policy_version 박기.

### [3] 가드레일 증적 뷰 (4-3) `P1` `🔴R9` `←2`
- **왜**: 적재된 증적을 부서·앱별로 조회·필터·상세. 증권사가 매일 볼 화면.
- **경쟁사는 이렇게**: Nutanix는 요청/엔드포인트 테이블 + 상세 페이지. Lakera는 위협 이벤트 목록. → 우리는 **trace_id·정책버전·마스킹샘플**까지 상세.
- **우리 구현 접근**:
  - 백엔드: `GET /api/v1/guard/audit?range=&decision=&type=` → ClickHouse `guard_audit` 쿼리(live provider 패턴 확장). 요약(검사/차단/PII/JB/flagged) + 행 목록.
  - 프론트: 새 화면(가드레일 nav 활성화) — 요약 카드 + 테이블(시각·부서·앱·유형·신뢰도·동작·상세) + 필터. SSOT 4-3 와이어프레임. Backend.AI 테이블 톤 재사용(usage-table).
- **검증**: 부하+가드레일 차단 생성 → 화면에 실데이터 목록·필터 동작, 상세 모달에 마스킹샘플(원문 비표시).
- **iteration 포인트**: WORM 원본 링크, CSV 내보내기, user_ref 드릴다운(권한 게이트).

### [4] rollup-worker (귀속 집계) `P1` `🔴R2·R3·R6`
- **왜**: 사용량 리포트의 **부서·앱·키 축**이 비어 있음(현재 모델축만). 메트릭 라벨에 user_ref를 못 붙이므로(카디널리티) **트레이스/메트릭 배치 롤업**이 정답(SSOT 3-1 원칙).
- **경쟁사는 이렇게**: Langfuse/Helicone은 요청 트레이스에서 user/app/cost를 집계. → 우리는 (user×app×key×model×시간) 5축.
- **우리 구현 접근**:
  - 소스: victoria-traces(OTLP) span attributes(`fabrix.app_id`/`api_key_id`/`model` + gen_ai.usage.*) 또는 dynamo_frontend 메트릭 + 키 매핑.
  - 적재: ClickHouse `usage_rollup`(SummingMergeTree, 이미 생성). 5분/1시간 버킷.
  - 구현체: Go CronJob/Deployment. live provider Usage()가 모델축은 vmselect로 이미 함 → 부서·앱축은 rollup 결과 조인.
- **검증**: 롤업 1회 실행 → 사용량 리포트에 dept/app 행이 실데이터로 채워짐.
- **iteration 포인트**: 트레이스에 fabrix 속성 주입(게이트웨이 단계), 버킷 크기, user_usage_rollup 분리(보존 짧게).

### [5] 키 쿼터(rpm/tpd) + 스펜드 `P1` `🔴R4` `※경쟁갭`
- **왜**: 키 발급·회수(✅)는 됐는데 **한도 강제·사용량 집계** 없음. 경쟁사 표준 기능.
- **경쟁사는 이렇게**: **LiteLLM** = 키별 budget/rpm/tpm 강제 + 스펜드 추적. **Portkey** = 키별 rate limit·예산. **Nutanix** = API Keys Top5(요청수). → 따라잡기 1순위.
- **우리 구현 접근**:
  - 스키마: `api_key`에 `quota_rpm`·`quota_tpd` 이미 있음(CNPG). 사용량은 usage_rollup의 api_key_id 축 조인.
  - 강제: 게이트웨이/프록시에서 키별 카운터(Redis 또는 메모리) 초과 시 429. dev는 우리 catalog 프록시에서 1차.
  - UI: 키·앱 화면에 쿼터 입력 + 키별 사용량/스펜드(Top5).
- **검증**: 쿼터 낮게 설정 → 초과 호출 429. 키별 사용량 표시.
- **iteration 포인트**: 분산 카운터(Redis), tpd 토큰 회계, 부서 단위 예산.

### [6] 워크로드 인클러스터 배포 `P1` `🔴R11`
- **왜**: 로컬 dev로 검증한 api·web을 클러스터에. DB도 인클러스터 직결(port-forward 불필요).
- **경쟁사는 이렇게**: 표준 GitOps. → 우리는 폐쇄망 전환 시 Harbor 전량 미러.
- **우리 구현 접근**: GitHub(maymustai) push → Actions(`build-and-push.yml`)→GHCR → `kubectl apply -f deploy/k8s/fabrix-endpoint.yaml`. api env: `FABRIX_DATABASE_URL=postgres://...@fabrix-pg-rw.fabrix-endpoint:5432/fabrix`, `FABRIX_VMSELECT_URL=http://vmselect-vm.observability:8481/...`, `FABRIX_GEMMA_UPSTREAM=http://gemma4-...-nodeport.dynamo-inference:8000`. web NodePort 30092.
- **검증**: NodePort 30092 접속 → 5개 화면 실데이터. (GHCR private면 imagePullSecret)
- **iteration 포인트**: Ingress/도메인, GHCR 가시성, secret 주입(CNPG app secret 마운트).
