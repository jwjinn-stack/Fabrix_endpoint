# observe — 배포 · 운영 · 검증 런북

> `FABRIX_PROFILE=observe` 배포의 실무 절차. 각 단계에 **왜 그렇게 하는지 근거**를 붙였다. 아키텍처 배경: [아키텍처.md](아키텍처.md).

## 1. 배포 단위
| 구성 | 무엇 | 비고 |
|---|---|---|
| **web** | nginx + 정적 SPA | `/api` 는 BFF 로 프록시(동일 오리진) |
| **backend(BFF)** | Go 단일 바이너리 | `FABRIX_PROFILE=observe` |
| (외부) 관측 스택 | Langfuse·ClickHouse·VM·Collector | **고객/제논이 제공** — FABRIX 는 인클러스터 ClusterIP 로 읽기 |

> **왜 2 Pod 만 배포하나** — observe 는 데이터를 만들지 않고 **읽기만** 한다. 관측 스택은 이미 고객/제논 클러스터에 있으니 FABRIX 가 또 세우지 않는다. Pod 를 클러스터 안에 두는 이유는 메트릭·k8s·트레이스에 **저지연·폐쇄망 내부 접근**을 하기 위함.

## 2. 환경변수 (읽기 소스 + 프로파일)
```bash
FABRIX_PROFILE=observe                 # 핵심: 읽기 전용
# 필요한 화면만 켜기(고객사별 가변)
FABRIX_FEATURES="+endpoints,+keys"     # 예: 엔드포인트/키 "읽기"도 노출(쓰기는 여전히 불가)
# 읽기 소스 (있는 것만 채움 — 없으면 그 화면만 비활성)
FABRIX_VMSELECT_URL=http://vmselect-vm.observability:8481/select/0/prometheus
FABRIX_CLICKHOUSE_URL=http://fabrix:<pw>@clickhouse.fabrix-endpoint:8123
FABRIX_LANGFUSE_HOST=http://langfuse-web.langfuse:3000
FABRIX_LANGFUSE_PUBLIC_KEY=pk-lf-...   # secret 은 Secret 으로
FABRIX_LANGFUSE_SECRET_KEY=sk-lf-...
FABRIX_HARBOR_URL=http://admin:<pw>@harbor-core.harbor
FABRIX_DATABASE_URL=postgres://...     # 마스킹 정책 조회용(있으면)
```
> **왜 creds 를 URL 에 두되 Secret 으로 주입하나** — 코드가 URL 파싱 후 creds 를 분리·헤더화하고 로그엔 호스트만 남긴다. 평문 노출을 막으려면 값 자체는 K8s Secret → env 주입.
> **왜 `FABRIX_FEATURES` 가 있나** — 삼성생명/협력사마다 "읽기로 보여줄 화면"이 다르다. 프로파일 기본(관제·사용량·가드레일·모델·트레이스)에 고객사별로 `+endpoints` 같이 가감한다. **읽기 cap 추가는 readonly 를 깨지 않는다**(write cap 은 별도).

## 3. RBAC (의도적으로 최소)
BFF ServiceAccount 에 **읽기 권한만**:
```
pods, pods/log: get, list, watch
DynamoGraphDeployment(CR): get, list, watch
```
> **왜 read-only RBAC 인가** — observe 는 어차피 mutating 라우트가 없지만, **RBAC 까지 읽기로 묶어** "이 SA 로는 클러스터를 못 바꾼다"를 인프라 레벨에서 한 번 더 보장한다(방어 심층화). create/delete 권한을 주지 않는다.

## 4. 연동 매칭 (고객 시스템 → FABRIX)
[../integration/README.md](../integration/README.md) 의 9개 의존성을 고객 실 시스템에 매핑한다. observe 는 전부 **읽기 엔드포인트**만 쓴다. 매핑이 끝나면 §6 진단으로 실제 도달성을 확인.

## 5. 검증 (배포 직후)
```bash
# (1) 프로파일이 읽기 전용인가
curl -s :8080/api/v1/capabilities | jq '.readonly, .profile'      # → true, "observe"
# (2) 쓰기가 정말 막혔는가 (구조적 차단 증빙)
curl -s -o /dev/null -w '%{http_code}\n' -XPOST :8080/api/v1/endpoints   # → 404
curl -s -o /dev/null -w '%{http_code}\n' -XPUT  :8080/api/v1/masking/policy # → 405
# (3) 읽기 소스가 실제 닿는가 (능동 프로브)
curl -s :8080/api/v1/diagnostics | jq '.checks[] | {name, configured, reachable, latency_ms, error}'
```
> **왜 (2) 를 검증하나** — "읽기 전용"이 말뿐이 아니라 **404/405 로 증명**됨을 배포마다 확인. 금융 감사·인수 테스트의 체크 항목.

## 6. 트러블슈팅 — `/diagnostics` 가 1차 도구
`reachable:false` + `error` 를 보고 원인을 좁힌다:
| 증상 | 흔한 원인 | 조치 |
|---|---|---|
| `connection refused` | 잘못된 host/포트, 서비스 미기동 | URL·svc·NetworkPolicy 확인 |
| `401/403` | creds 불일치 | Secret 값·키 권한 확인 |
| `configured:false` | env 미설정 | 해당 `FABRIX_*` 주입(없으면 그 화면 비활성은 정상) |
| 화면만 빈 상태 | 소스는 닿으나 데이터 0 | 고객 측 적재 파이프라인(Collector/로그) 확인 |

## 7. 마스킹 정책 출처 (반드시 결정)
글루가 원문을 캡처한다면 정책이 필요하다. observe 는 편집 불가(PUT 405)이므로 출처를 택1:
1. **기본 정책 사용**(아무 설정 없이 글루 기본값: 입출력 마스킹·차단건 원문·민감 PII 해시) — 가장 간단.
2. **PG 시드** — `masking_policy` 단일 행을 운영자가 1회 삽입.
3. **manage 인스턴스로 설정** 후 observe 가 읽음.
> **왜 함정인가** — observe 만 띄우면 "마스킹을 바꿀 화면이 없다". 사전에 출처를 정해두지 않으면 글루가 기본값으로 동작하는 줄 모르고 원문 노출 범위를 오해할 수 있다.

## 8. NetworkPolicy (폐쇄망)
> **왜** — 금융 폐쇄망은 "필요한 통신만 허용"이 원칙. BFF→(VM/ClickHouse/Langfuse/Harbor/k8s read) egress 만 열고, 그 외는 막아 데이터 유출 경로를 좁힌다. Langfuse 가 셀프호스트면 클라우드로의 송신이 없어야 한다(데이터 레지던시).
